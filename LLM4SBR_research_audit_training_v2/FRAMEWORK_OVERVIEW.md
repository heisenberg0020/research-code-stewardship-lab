# Research Code Stewardship Lab

## 用代码审计训练 Coding Agent 时代的研究判断力

这是一个面向“论文 + 源码 + 实验”任务的研究代码审计训练仓库。它不训练人把代码写得更快，
而训练人识别那些**能够编译、能够运行、甚至能够产出看似合理结果，却已经破坏论文科学含义**的细微错误。

仓库以 LLM4SBR（Session-based Recommendation with Large Language Models）为首个完整案例，
提供论文与原始实现的正式来源链接、四级候选代码题、答案隔离材料、验证脚本，以及一个可以迁移到其他论文的
`research-code-audit-training` Skill。它适合用来训练研究程序员、研究生、研究工程师，以及负责监督
Coding Agent 的实验负责人。

### LLM4SBR 论文与原始实现

本仓库的首个训练案例来自论文 **LLM4SBR: A Lightweight and Effective Framework for Integrating
Large Language Models in Session-based Recommendation**。为尊重原作者的研究贡献，论文和原始代码均
保留其正式来源，并与本仓库的训练材料明确区分：

- [论文：LLM4SBR（arXiv:2402.13840）](https://arxiv.org/abs/2402.13840)
- [原始代码仓库：tsinghua-fib-lab/LLM4SBR](https://github.com/tsinghua-fib-lab/LLM4SBR)
- [论文学习协议生成的参考样例：LLM4SBR_PAPER_STUDY_GUIDE.md](docs/examples/LLM4SBR_PAPER_STUDY_GUIDE.md)

本仓库不主张替代论文或原始实现，也不把训练候选中的任何实现选择归因于作者。LLM4SBR 原始材料仅
作为论文复现、代码审计和人类能力训练的研究对象；论文结论、作者署名和原始代码的版权与贡献归原作者
及其发布渠道所有。开始训练前，建议先阅读正式论文和原始仓库说明。

### 这个仓库为什么存在

Coding Agent 已经能够完成大量框架搭建、模块实现、测试、训练和实验运维工作。真正稀缺的能力因此
从“能不能写出代码”转向“能不能判断代码是否忠实、实验是否公平、证据是否足够、Agent 是否越权”。
本仓库把这些判断拆成可重复、可验证、逐级增加上下文的训练题，帮助人类保留研究协议、科学解释和
最终主张的控制权。

### 从哪里开始

1. 先阅读本文的四级框架和 Coding Agent 能力边界；
2. 再看 [REPOSITORY_MAP.md](REPOSITORY_MAP.md) 了解公开材料与隔离材料的位置；
3. 如果要先学习论文，参考 [LLM4SBR_PAPER_STUDY_GUIDE.md](docs/examples/LLM4SBR_PAPER_STUDY_GUIDE.md) 的结构和 `PRE_AUDIT_BASELINE`；
4. 再使用 [四级训练包](LLM4SBR_research_audit_training_v2/README.md) 做题，并按答案模板记录定位、依据、反例、因果影响和修复建议；
5. 若要为另一篇论文生成同类训练，阅读并安装 [Research Code Audit Training Skill](skills/research-code-audit-training/SKILL.md)。

本文后续章节是可公开阅读的通用导论：第 0 节介绍 Skill，第 1–3 节说明训练目标、Agent 能力和
程序员角色，后续章节定义四个 Level 的错误家族、证据标准、实施流程和验收门槛。公开文档不包含
具体题目的候选答案、实际变异点或隐藏测试；教师版答案与探针保持隔离。

## 0. 使用 Research Code Audit Training Skill

### 0.1 Skill 的功能与训练目标

`research-code-audit-training` 不是一个新的模型，也不是替代 Codex 的独立程序；它是一组供 Coding
Agent 遵循的专业工作规范。它把一篇论文和其源代码转换为可验证的“四级盲审训练包”：每个 Level
提供 5 份接口一致、能够完成受限运行的候选实现，其中 1 份忠实实现论文，另外 4 份各包含一个主要的
隐蔽错误。

四个 Level 对应四种不同的审计对象：

1. **算法语义**：公式、算子、索引、归一化和梯度是否实现正确；
2. **流水线完整性**：数据身份、切分、排列、checkpoint 和评估状态是否保持一致；
3. **科学有效性**：比较预算、信息条件、实验单位、排除规则和证据是否支持结论；
4. **Agent 实验治理**：权限、审批、资源预算、失败记录和受保护证据是否被遵守。

因此，训练目标不是发现 syntax error，而是判断“一个能运行的实验系统是否仍然代表论文声称的科学
实验”。Agent 负责阅读、构建、变异、测试和整理材料；人负责确认论文解释、冻结实验协议、批准设计，
并根据证据判断候选是否可信。Skill 会先生成设计规格，只有在人工确认后才进入候选实现阶段。

### 0.2 输入、目录与安全边界

使用 Skill 时，你需要提供以下输入。它们分别回答“研究什么”“从哪里实现”“把训练包写到哪里”以及
“哪些资源不能动”：

1. **论文**：PDF、正文或可读取的论文路径；
2. **源代码**：作者仓库或本地源码目录；
3. **输出目录**：一个新的、允许写入的训练包目录；
4. **运行约束**：可用的 Python/PyTorch 环境、smoke test 预算、GPU/数据限制；
5. **保护清单**：不得修改、上传、联网访问或完整运行的文件和资源。

你只需要具备基本的文件路径和终端操作能力，不必先掌握完整的 PyTorch 或论文领域知识；但应能阅读
Skill 输出的设计规格并确认实验规则。论文、原始源码和数据默认只读，候选代码、答案映射和隐藏探针
必须写入训练包的相应目录，不能覆盖原始研究材料。

如果论文与源码对算法或实验协议的描述不一致，Skill 只能报告冲突和证据，不能擅自把源码当作“正确
答案”。应先冻结采用的定义、切分、指标和比较协议，再继续生成题目。

### 0.3 本仓库里 Skill 文件在哪里

- [Skill 主文件](skills/research-code-audit-training/SKILL.md)：触发条件、完整工作流、TDD 顺序、公开与隐藏材料隔离规则；
- [四级能力与错误边界](skills/research-code-audit-training/references/four-level-framework.md)：Level 1–4 的归类规则和通用错误家族；
- [Level 3/4 实施经验](skills/research-code-audit-training/references/level3-level4-implementation-lessons.md)：科学 dossier 与 Agent 治理时间线的结构化证据要求；
- [质量门与修正方法](skills/research-code-audit-training/references/quality-gates.md)：单一错误、反猜题、误报控制及 RED/GREEN 审核要求；
- [训练包产物契约](skills/research-code-audit-training/references/artifact-contracts.md)：目录结构、manifest、ledger、claim 和验证入口；
- [跨领域适配指南](skills/research-code-audit-training/references/domain-adaptation.md)：将 session、candidate、checkpoint 等概念映射到医疗、图学习、强化学习等领域；
- [设计规格模板](skills/research-code-audit-training/assets/design-spec-template.md)：实施前交给用户确认的四级题目规格；
- [学员答案模板](skills/research-code-audit-training/assets/answer-sheet-template.md)：要求提交定位、依据、反例、因果影响和修复；
- [训练包静态验证器](skills/research-code-audit-training/scripts/validate_training_package.py)：检查四级结构和公开材料泄题风险。

### 0.4 安装到 Codex

如果你使用 Codex，打开终端，进入本仓库的根目录，执行下面两行命令。它们会把 Skill 复制到
Codex 的 Skill 文件夹中：

```bash
mkdir -p ~/.codex/skills
cp -R skills/research-code-audit-training ~/.codex/skills/
```

安装完成后，在新的 Codex 对话中用下面这个名称调用它：

```text
$research-code-audit-training
```

如果 Codex 提示已经有同名 Skill，请先删除旧版本或用本仓库版本覆盖它，避免出现“Skill 套 Skill”
的重复目录。本仓库版本已经包含 Level 3 和 Level 4 在实际验证中发现的修正规则。

### 0.5 第一次使用：直接复制这段话

把下面的文字复制到 Codex，并把三条路径换成你自己的路径：

```text
使用 $research-code-audit-training，根据下面的论文和源代码设计四级研究代码审计训练。

论文：/你的路径/paper.pdf
源码：/你的路径/source-repository
输出：/你的路径/training-package

请按以下方式工作：
- 先阅读论文和源码，列出两者的对应关系和冲突，不要直接修改原始文件；
- 先给我一份四级题目设计方案，等我确认后再生成题目；
- 每个 Level 生成 5 份都能运行的候选代码，其中 1 份可信，4 份各有一个隐蔽错误；
- 用自动测试、隐藏检查和回归测试确认题目有效；
- 把答案、错误位置和隐藏测试放在教师专用目录，不要放进学员可见材料；
- 不联网、不运行完整训练，不修改论文和原始源码。
```

如果你还不想生成代码，只想先讨论方案，可以使用：

```text
使用 $research-code-audit-training，只完成论文到代码的对应关系、实验规则冻结和四级题目设计方案，
等待我确认后再生成任何候选代码。
```

### 0.6 生成后你会得到什么

生成完成后，输出文件夹通常包含：

- 四个 Level 的题目说明和学员答案表；
- 每个 Level 的 5 份候选代码或实验材料；
- 可重复运行的公开检查和总入口；
- 只供教师使用的答案、错误定位、最小修复和隐藏验证；
- 设计规格、测试过程、RED/GREEN 记录和已知限制。

做题时不要只回答“我选 A”。请说明：哪一份可信、其他候选违反了论文或实验协议的哪条规则、为什么
它仍然能够运行、如何用最小反例证明问题，以及最小安全修复是什么。这样训练的是研究判断能力，
而不是猜字母。

### 0.7 Skill 会怎样保证题目质量

Skill 应按以下顺序工作：

1. 核对论文、源码、环境、保护路径和资源边界；
2. 建立论文公式、代码位置、实验协议与可执行不变量的映射；
3. 输出四级设计规格并等待人工确认；
4. 逐级执行测试先行的 RED → 实现 → GREEN → 审核 → 修正；
5. 生成学生可见材料、隔离答案、隐藏探针和实施报告；
6. 执行公开检查、包外隐藏验证、重复性、泄题、旧文件哈希和发布检查；
7. 只在全部证据通过后声明训练包完成。

Skill 不会替人决定论文创新是否成立，也不会保证实验获得正结果。它负责把出题与验证流程
标准化；人仍然负责冻结科学协议、批准关键改变、判断证据强度和承担最终主张责任。

### 0.8 四级训练的前置阶段：先不看源码，建立独立的论文理解

这一步位于论文阅读与代码训练之间：学习者先只依据论文建立判断标准，再进入 Level 1 首次查看候选
代码。它解决的是一个关键偏差——如果先看作者仓库，学习者很容易把“代码当前怎么写”误当成“论文
必然要求怎么写”。

完整规则位于
[PAPER_ONLY_REPRODUCTION_PROTOCOL](docs/PAPER_ONLY_REPRODUCTION_PROTOCOL.md)。协议提供两个模式：

| 模式 | 何时使用 | 输出 |
|---|---|---|
| **A：Source-Blind Pre-Audit Learning（默认）** | 开始四级训练前；即使作者源码存在，也主动隔离 | `PAPER_STUDY_GUIDE.md` |
| **B：Paper-Only Clean-Room Reproduction** | 作者没有源码，并且需要进一步规划独立复现 | `PAPER_REPRODUCTION_GUIDE.md` |

本仓库的标准学习路线使用模式 A。这里的 source-blind 表示 Agent 不读取作者源码、第三方实现、A–E
候选、答案、hidden probes 或相关 Git 历史。指南可以使用公式、tensor contract、系统图和解释性
伪代码，但这些只能标为论文证据 `[P]`、推导 `[I]` 或假设 `[A]`，不能被描述成作者实现。

#### 0.8.1 建立隔离的论文学习目录

新建一个独立目录，只放入：

1. 论文 PDF；
2. appendix 或 supplementary material；
3. 必要的数据集说明；
4. [PAPER_ONLY_REPRODUCTION_PROTOCOL.md](docs/PAPER_ONLY_REPRODUCTION_PROTOCOL.md)。

不要把 `LLM4SBR-main/`、`LLM4SBR_research_audit_training_v2/`、候选文件或
`DO_NOT_OPEN_UNTIL_FINISHED/` 放入这个目录。最可靠的隔离方式是用该目录新开一个 Codex 任务，使
Agent 的工作区根目录本身就不包含训练材料。

#### 0.8.2 复制 Prompt 并替换路径

下面的 Prompt 是本仓库推荐的模式 A 入口。将三处路径替换为你的实际路径后发送给 Agent：

```text
请完整阅读并严格遵循：
/absolute/path/to/paper-study/PAPER_ONLY_REPRODUCTION_PROTOCOL.md

使用模式 A：Source-Blind Pre-Audit Learning。

允许读取的唯一根目录：/absolute/path/to/paper-study
主论文：/absolute/path/to/paper-study/paper.pdf
输出文件：/absolute/path/to/paper-study/PAPER_STUDY_GUIDE.md

本轮目标是在首次查看代码之前，仅依据论文、附录、补充材料和数据说明，建立四级研究代码审计所需的
独立论文基准。即使其他工作区存在作者源码或训练题，也不得搜索、读取、引用或推断以下内容：作者源码、
第三方实现、A–E 候选、答案映射、DO_NOT_OPEN_UNTIL_FINISHED/、hidden probes、相关测试或 Git 历史。

请在同一个 PAPER_STUDY_GUIDE.md 中完整交付：
- Unit 1–5；
- Unit 1 与 Unit 2 之间的工程化大框架；
- [P]/[I]/[A]/[E] 证据标记和统一假设账本；
- PRE_AUDIT_BASELINE；
- Level 1–4 衔接表和进入 Level 1 前准备度清单。

重点覆盖任务与贡献、端到端数据流、关键公式、tensor shape、运算轴、mask、ID/target 空间、loss、
gradient path、数据切分、checkpoint/test 边界、评价指标、实验比较条件和论文 claim 范围。

不要生成项目源码，不要运行训练，不要判断候选答案。解释性伪代码和工程图不得冒充作者实现；论文未
规定的内容必须保留为可替换的 [A]，不能补写成唯一正确实现。完成后重新检查指南结构，聊天中只返回
完成状态和文件链接。
```

如果希望学习材料完全不出现伪代码，可在 Prompt 末尾追加：

```text
本次使用 strict no-code 变体：不输出任何 Python/PyTorch 代码块，只保留公式、tensor contract、
shape 表、数据流图、自然语言算法步骤和审计不变量。
```

#### 0.8.3 检查输出是否可以用于 Level 1

不要只确认文件已经生成。开始 Level 1 前，检查 `PAPER_STUDY_GUIDE.md` 是否同时包含：

- 五个完整 Unit 和独立的工程化大框架；
- 可追溯到论文位置的 `[P]`，以及与其分开的 `[I]` 和 `[A]`；
- 关键公式的 shape、axis、mask、target、loss 和 gradient contract；
- 数据切分、candidate set、metric、checkpoint 和 test 使用边界；
- `PRE_AUDIT_BASELINE` 中的 claim、算法、数据/评估、实验比较和不确定性清单；
- Level 1–4 衔接表，并明确 Level 4 仍需额外的审批、预算、停止和记录协议；
- 没有候选字母、答案映射、源码位置或 hidden rule。

模式 A 到这里结束。学习者能够用自己的话解释 `PRE_AUDIT_BASELINE` 中的核心契约后，再使用第 0.9
节的 Prompt 进入 Level 1。后续判断必须以 `[P]` 和必要的 `[I]` 为依据；某个候选仅仅不同于 `[A]`，
不能因此判错。

### 0.9 给 Agent 的“训练从哪里开始”Prompt

如果学习者不知道应该从哪个 Level 开始，可以把下面的 Prompt 直接发送给 Agent。它要求 Agent 先判断
学习者的前置知识和当前材料状态，再给出可执行的起步路径；不会直接透露候选答案。

```text
我准备开始这个仓库中的四级研究代码审计训练。请你作为学习向导，而不是直接替我做题。

请先根据我提供的论文、源码、训练包路径和我的背景，判断我是否已经具备开始当前 Level 所需的最低知识。
如果不足，请只补充开始该 Level 必需的概念，不要讲完整课程，也不要提前揭示任何候选答案、错误位置或隐藏规则。

然后请按下面格式告诉我如何开始：
1. 我当前应该从 Level 1、2、3 还是 4 开始，以及原因；
2. 需要先阅读的文件，按先后顺序列出；
3. 需要运行的公开命令；
4. 本关要审计的对象和最容易混淆的概念；
5. 我应该在 ANSWER_SHEET.md 中记录哪些证据；
6. 完成后用什么标准判断可以进入下一关。

约束：
- 不打开或引用 DO_NOT_OPEN_UNTIL_FINISHED/ 下的答案、manifest 和 hidden probes；
- 不根据候选字母、代码长度、注释风格或输出数值猜答案；
- 不直接告诉我哪一份候选可信；
- 如果我的信息不足，先列出缺少的路径或背景，再给出基于当前信息的最小可行起步方案；
- 教学目标是训练“代码能运行但科学语义可能错误”时的审计能力。
```

## 1. 为什么需要这套训练

现代 Coding Agent 已经可以承担大量代码阅读、框架搭建、测试、运行和修复工作。问题不再
只是“Agent 能不能写出代码”，而是：

> 当代码能编译、训练能运行、loss 会下降、结果表看起来合理时，人是否仍能判断这套系统
> 实现了正确的问题、遵守了正确的实验协议，并支持它声称的科学结论？

研究代码最危险的错误通常不会制造编译失败。它们可能只改变一个归一化维度、约束边界、
梯度路径、数据切分单位或模型选择条件，却足以让训练目标、评估结果或论文结论失效。

因此，本框架不把训练目标定义为“比 Agent 更快写代码”，而是培养四类不可缺少的审计
能力：

1. 判断局部算法是否忠实实现数学定义；
2. 判断数据、训练和评估流水线是否保持身份与状态一致；
3. 判断实验比较和论文主张是否公平、完整、有足够证据；
4. 判断 Agent 的行动是否遵守权限、预算、审批和记录要求。

## 2. 当前 Coding Agent 可以达到什么能力

### 2.1 官方能力定位

OpenAI 将 GPT-5.6 Sol 定位为适合复杂编码、研究、计算机操作和高价值开放任务的旗舰
模型。Codex 的 Ultra 模式不是另一个独立模型，而是让所选模型使用子智能体并行处理可
拆分的复杂任务。官方说明，多智能体可能降低墙钟时间，也可能改善适合独立拆分工作流的
复杂任务表现；同时，大多数普通任务并不需要 Max 或 Ultra。

参考：

- [Codex Models](https://learn.chatgpt.com/docs/models)
- [Codex Subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents)
- [GPT-5.6 model guidance - What is new](https://developers.openai.com/api/docs/guides/latest-model#what-is-new)

这些定位说明 Coding Agent 的能力上限很高，但不构成“任何研究任务有 99% 正确率”或
“固定可替代 80%-90% 人力”的保证。可靠比例必须通过与真实任务同分布的盲测、失败率、
误报率和长程运行记录来估计。

### 2.2 在条件明确时，Agent 擅长的工作

当输入材料、权限和验收标准明确时，成熟 Coding Agent 通常适合承担：

- 阅读论文、代码、配置、日志和数据说明；
- 按冻结规格搭建项目结构、模块接口和训练入口；
- 实现数据加载、模型、loss、优化器、评估与 checkpoint；
- 编写单元测试、集成测试、shape 检查、NaN/Inf 检查和梯度检查；
- 运行 smoke test、baseline、消融、多随机种子和批准范围内的参数搜索；
- 监控训练、恢复中断、整理结果、绘图和生成实验摘要；
- 在彼此独立的模块、实验或审查任务间使用子智能体并行工作；
- 根据明确失败证据修复普通实现、依赖和环境问题。

这些工作有一个共同特征：目标、约束、输入和“完成”的定义可以被写成可检查规格。
官方建议优先把子智能体用于探索、测试、分诊和日志分析等读密集任务；并行写密集工作
需要更谨慎，因为会增加冲突和协调成本。

### 2.3 Agent 不能自动保证的事情

即使 Agent 能完整交付工程系统，也不能由此自动保证：

- 论文问题定义本身合理；
- 数据划分没有泄漏；
- baseline 获得了公平的输入、预算和调参机会；
- 指标真正衡量研究问题；
- 结果没有受到 seed、checkpoint 或 test peeking 的选择偏差；
- 提升具有统计意义和实际意义；
- 负结果、失败运行和异常样本被完整保留；
- 论文创新点、因果解释和最终主张成立。

Agent 可以协助检查这些问题，但最终可信度不能只由“执行同一任务的 Agent 自己检查自己”
来建立。

同样，测试全部通过只表示测试中编码的不变量成立；如果规格、数据协议或评价定义本身
错误，测试可能完整验证一个错误系统。

## 3. 程序员在代码框架构建中的新角色

Coding Agent 提高的主要是执行带宽，不会消除工程和研究责任。程序员的角色会从“每一行
都亲自输入”转向以下五种角色。

### 3.1 系统架构师

负责确定：

- 模块边界、数据契约和接口；
- 哪些组件可以独立替换；
- 训练、评估和报告之间的数据流；
- 哪些不变量必须由测试机械执行；
- 哪些异常必须停止而不是自动修复。

### 3.2 规格与实验协议所有者

负责冻结：

- 数据版本、样本构造和 split；
- 模型、loss、候选集和评价指标；
- baseline 配置和调参预算；
- checkpoint 选择规则；
- test set 使用次数；
- 随机种子、重复次数和统计方法。

### 3.3 风险与权限审批人

负责区分：

- Agent 可以自行完成的常规实现；
- 可以在预先批准范围内调整的参数；
- 必须在改变前请求批准的数据、模型、指标、预算和研究假设；
- 无论结果如何都不得执行的泄漏、删记录和事后改协议行为。

### 3.4 证据审计员

负责要求每个判断都能追溯到：

- 论文公式或冻结规范；
- 代码行、配置键、run_id、event_id；
- 可重复的最小反例；
- 原始日志和实验账本；
- 能证伪当前实现的测试，而不只是主观代码风格判断。

### 3.5 科学主张把关人

负责决定：

- 哪些结果可以报告；
- 哪些只能称为观察，不能称为稳定提升；
- 哪些差异需要补跑实验；
- 哪些证据支持相关性但不支持因果解释；
- 最终论文主张是否超过数据允许的强度。

一句话概括：

> Agent 负责高带宽执行，程序员负责规格、边界、证据和最终责任。

## 4. 四个 Level 的核心定义

四个 Level 不是线性难度梯子。随着 Level 提升，审计范围从局部计算，扩展到系统集成、
科学证据和全过程治理：

| Level | 主要审计范围 | 核心问题 |
|---|---|---|
| Level 1 | 一个算子或局部函数 | 数学目标是否被忠实实现？ |
| Level 2 | 多文件训练流水线 | 样本、状态和评估是否始终一致？ |
| Level 3 | 整组实验与论文结论 | 比较是否公平，主张是否被证据支持？ |
| Level 4 | Agent 行动与治理过程 | 谁批准了改变，过程是否合规且可审计？ |

Level 4 虽然编号最后，却不是只发生在项目末尾。它是覆盖需求、数据、模型、训练、评估和
报告全过程的横向治理层。

当同一个问题会造成多层后果时，按“首先被破坏的契约”确定唯一主 Level：

- 局部公式或算子语义首先错误，归为 Level 1；
- 流水线自动执行中的数据流、身份、状态或评估契约首先错误，归为 Level 2；
- 比较设计、证据汇总或论文推断规则首先错误，归为 Level 3；
- Agent 的自适应行动未经授权，或违反审批、预算、记录边界，归为 Level 4。

下游影响可以跨 Level，但每道题只设置一个主要错误和一个主 Level，避免同一错误被重复
计分。

为适配不同研究范式，本文中的术语按论文协议解释：

- “测试集”泛指受保护的最终评价证据，不限于监督学习中的静态 test split；
- “切分实体”是论文定义的独立单位，可以是用户、会话、时间段、患者、图或环境实例；
- “候选集”泛指评价总体或决策空间；
- “checkpoint”泛指被选择并进入最终评价的模型或实验产物；
- 在强化学习中，还须冻结环境版本、终止语义、交互数据边界和策略选择规则。

### 4.1 四级题目、框架能力与错误家族总表

| Level | 学员面对的材料 | 对应框架能力 | 主要错误家族 | 首选证明方式 |
|---|---|---|---|---|
| 1 | 五份接口一致的局部实现 | 公式到张量、loss、索引、梯度 | 运算域、约束、目标输入、更新路径 | 性质测试、极小反例、数值或梯度检查 |
| 2 | 五套均可运行的多文件流水线 | 数据身份、状态生命周期、评估协议 | 隔离、对齐、聚合、选择与恢复 | 数据血缘、联合身份、独立重算、事件顺序 |
| 3 | 五份完整实验 dossier | baseline、预算、统计、报告 | 公平性、信息条件、选择性报告、推断不足 | 配置对照、账本重建、预注册统计规则 |
| 4 | 五份 Agent 运行与治理记录 | 权限、审批、资源、日志、停止条件 | 越权、超范围、记录破坏、受保护证据适应 | 协议逐条核对、时间线与审批链审计 |

四级均采用“一份可信、四份各含一个主要错误”的盲审形式，但被审查对象逐步从代码片段
扩展到流水线、科学证据包和 Agent 行动记录。

## 5. Level 1：算法语义正确性

### 5.1 对应的框架能力

Level 1 对应代码框架中的模型、loss、注意力、检索、归一化、索引和梯度路径。它训练：

- 从论文公式建立张量 shape 和语义表；
- 区分“shape 合法”和“数学正确”；
- 检查广播、reduction axis、mask、normalization、logits 和 probability；
- 追踪 autograd，确认辅助目标真正影响应训练的参数；
- 使用不变量和极小张量，而不是等待完整训练指标恶化；
- 证明一行代码如何改变优化目标。

### 5.2 Level 1 关注的错误点

Level 1 专门使用微型、高危、非编译错误，主要错误家族包括：

- **运算域与维度语义**：张量 shape 合法，但归一化、聚合或广播发生在错误语义维度；
- **有效域与约束边界**：无效位置、非法类别或边界元素悄然参与计算；
- **目标函数输入契约**：分数、概率、尺度、符号或权重与损失函数的数学假设不一致；
- **梯度与更新路径**：数值和日志仍正常，但目标无法更新应训练参数，或更新了不应训练参数；
- **索引与身份语义**：索引仍在合法范围内，却指向错误类别、时间步或特殊标记；
- **缩放与运行状态**：聚合尺度、随机层或推理状态改变了算法在不同 batch 和阶段下的含义。

好的 Level 1 错误应满足：

- 改动通常只有一个参数、常数、操作或逻辑行；
- 所有接口、shape、forward 和 backward 都可执行；
- 输出有限且可能与正确实现拥有相同 argmax；
- 单次 loss 大小不能直接暴露答案；
- 必须通过数学不变量、有限差分或梯度探针证明。

### 5.3 Level 1 不应使用的错误

- 变量未定义、括号错误和 import 失败；
- 明显 shape mismatch；
- 删除完整模块；
- 一眼可见的 long/short 整体接反；
- 只有运行数小时后才知道“指标似乎不好”，却没有局部证据；
- 通过五份文件的多数写法或机械 diff 就能拼出正确答案。

## 6. Level 2：数据、训练与评估流水线完整性

### 6.1 对应的框架能力

Level 2 对应数据预处理、DataLoader、shuffle、训练循环、checkpoint、scheduler、评估器和
日志系统。它训练：

- 追踪同一个样本在多文件和多阶段中的身份；
- 区分原始实体、派生样本和 batch 索引；
- 验证输入、标签、语义特征和 mask 使用同一 permutation；
- 检查 train/validation/test 的角色边界；
- 检查 optimizer、scheduler、随机数和 checkpoint 状态是否完整恢复；
- 从逐样本结果重算聚合指标；
- 使用 provenance 和事件日志发现 test peeking。

### 6.2 Level 2 关注的错误点

通用错误家族包括：

1. **数据边界与隔离失效**：派生数据、缓存或预处理顺序破坏了冻结的独立性边界；
2. **样本身份与跨模态对齐失效**：多个仍然合法的张量、标签或特征不再描述同一对象；
3. **评估总体与聚合契约失效**：样本集合、分母、目标位置或过滤规则偏离冻结定义；
4. **模型选择与状态生命周期失效**：受保护证据参与选择，或恢复后的运行不再等价于连续运行；
5. **阶段间处理不一致**：训练、验证、最终评价使用了未经声明的不同映射或数据语义。

这些错误可能让训练曲线正常，甚至让指标更好。Level 2 的证据通常需要跨越多个文件，不能
只看模型 `forward()`。

## 7. Level 3：实验公平性与科学有效性

### 7.1 对应的框架能力

Level 3 对应配置管理、baseline、搜索预算、重复实验、结果聚合、统计分析和论文报告。它
训练：

- 判断比较对象是否使用相同数据、特征、候选集和计算预算；
- 判断 baseline 是否获得公平调参；
- 从所有按预定协议应纳入的运行重建结果，并披露预先规定的排除项；
- 区分均值提升、稳定提升、显著提升和实际重要提升；
- 判断 ablation 是否真正隔离一个变量；
- 将论文主张限制在证据允许的范围内；
- 明确需要补跑的最小实验。

### 7.2 Level 3 关注的错误点

通用错误包括：

1. **baseline 不公平**：新模型获得更大预算、更好调参或更多重试；
2. **信息条件不一致**：一方使用额外特征、预训练数据、候选集或后处理；
3. **选择性报告**：挑最好 seed、最好数据集、最好指标或最好 checkpoint；
4. **统计推断不足**：未按预先规定的独立或配对实验单位进行适当不确定性分析，或效应量、
   区间与多重比较处理不足以支持所声称的显著、稳定或普遍优越；
5. **多重比较未控制**：搜索大量组合后只报告赢家；
6. **消融混杂**：一次删除多个组件，却将差异归因于其中一个；
7. **指标替换**：结果不好后改用更有利指标，而未预先登记；
8. **复现条件缺失**：数据版本、代码 SHA、依赖或 seed 无法追溯。

Level 3 中可能出现“所有代码都正确，但论文结论仍不可信”。这正是它与 Level 1/2 的根本
区别。

随机种子并不自动等于相互独立的统计样本。出题前必须冻结实验单位、配对关系、重复结构和
允许使用的推断方法；学生应依据这份协议判断主张，而不是只观察两组边际区间是否重叠。

## 8. Level 4：Agent 权限与科学实验治理

### 8.1 对应的框架能力

Level 4 对应任务规格、审批边界、资源预算、实验账本、Agent 事件日志和最终报告。它训练：

- 区分技术上可执行与权限上被授权；
- 判断改变发生前是否获得批准；
- 检查 Agent 是否在关键歧义处停止并请求人工决策；
- 验证失败运行和负结果是否完整保留；
- 检查最终报告能否由原始账本重建；
- 区分工程异常、科学风险、权限越界和研究伦理问题；
- 对 Agent 行动作 `approve`、`block` 或 `needs-human-approval` 决策。

### 8.2 Level 4 关注的错误点

通用错误家族包括：

1. **权限边界偏移**：Agent 自行改变本应由人批准的研究对象、数据、模型、指标或假设；
2. **范围与资源越界**：执行超出批准搜索空间、运行次数、算力或停止条件的行动；
3. **记录完整性破坏**：失败、负结果、关键配置或行动记录被删除、覆盖或选择性汇报；
4. **受保护证据驱动适应**：最终评价信息未经披露地驱动后续开发或选择，或将事后形成的
   主张伪装成预先假设；
5. **审批状态误读**：将待定、沉默或范围有限的批准解释成全面授权；
6. **多 Agent 协议漂移**：子任务执行了彼此不一致或未经批准的科学协议。

正确的 Agent run 不一定拥有最高指标。它可能因为超出权限而停止、保留负结果，或明确请求
人工批准。Level 4 必须避免让学生把“结果最好”误认为“过程最可信”。

Level 4 题目必须提供封闭、完整、按时间排序的审批记录与事件日志，或者提供能够正面证明
行动与批准冲突的证据。仅凭“日志里没找到批准”不能推断现实中从未批准；若材料完整性没有
被冻结，正确结论应是证据不足，而不是直接判定越权。

## 9. 四个 Level 与代码框架的直接映射

| 框架环节 | Coding Agent 可承担 | 程序员必须控制 | 主要 Level |
|---|---|---|---|
| 问题与需求 | 整理需求、生成规格草案 | 冻结研究问题、歧义与成功标准 | 3、4 |
| 数据获取与切分 | 编写下载、清洗、切分代码 | 确认数据合法性、泄漏边界和 split 单位 | 2、3、4 |
| 特征与样本构造 | 实现编码、缓存、对齐 | 确认没有未来信息和模态错位 | 2 |
| 模型与 loss | 实现模块、forward、梯度测试 | 验证公式、维度、优化目标 | 1 |
| 训练编排 | 运行训练、恢复、记录资源 | 冻结预算、停止条件和允许搜索空间 | 2、4 |
| checkpoint | 保存和恢复状态 | 确认选择只使用协议允许的开发或验证证据，不接触受保护的最终评价证据 | 2、4 |
| 评估指标 | 实现与批量计算 | 冻结候选集、分母、过滤和统计单位 | 2、3 |
| baseline 与消融 | 批量运行批准实验 | 保证信息、预算和调参公平 | 3、4 |
| 结果统计 | 汇总 seed、生成图表 | 判断显著性、实际意义和多重比较 | 3 |
| 报告与论文 | 生成表格和文字草案 | 决定允许主张、限制和伦理披露 | 3、4 |
| Agent 协作 | 分派子智能体、汇总结果 | 防止协议漂移、越权和证据丢失 | 4 |

这个映射说明：四级训练不是附加在编程之外的理论课程，而是直接覆盖代码框架从需求到论文
结论的整个生命周期。

## 10. 通用题目结构

为了让同一训练方式适用于不同论文，每一级采用统一形式：

1. 提供论文、源码、冻结规范或证据材料；
2. 提供 A-E 五份外观合理的候选，其中一份完全可信；
3. 四份错误候选各包含一个主要目标错误；
4. 所有候选通过公开的编译、schema 或 smoke test；
5. 公开检查只证明“材料可用”，不证明科学正确；
6. 学生必须提交错误位置、依据、影响和可重复证据；
7. 答案、隐藏探针和评分示例放入隔离目录；
8. 正确候选位置在不同论文和 Level 间变化；
9. 文件长度、注释、命名和输出格式不得泄漏答案；
10. 完成后用隐藏探针验证每个错误恰好被目标证据稳定识别。

## 11. 通用出题原则

### 11.1 错误必须真实

错误应来自真实研究风险，而不是人为谜语。每个错误都应回答：

- 现实开发中为什么可能出现？
- 为什么普通 smoke test 抓不到？
- 它违反哪条公式、数据契约、实验协议或审批规则？
- 它会造成什么风险或影响？若不能从现有证据确定偏差方向，是否明确标为未知？
- 什么最小证据可以稳定证明？

### 11.2 错误必须隐蔽但可证伪

“隐蔽”不等于无解。每道题必须存在比完整训练更便宜、更明确的证据方法，例如：

- 性质测试、变形测试或局部数值验证；
- 身份、集合、顺序与数据血缘审计；
- 从原始记录独立重算中间量或最终指标；
- 论文协议、冻结配置与实际运行配置的一致性检查；
- 按时间顺序核对行动、审批、资源和实验账本；
- 从账本重建最终报告并检查是否存在遗漏。

### 11.3 不允许机械猜题

- 正确候选不能总是最短、最整洁或位于同一字母；
- 不得让错误候选拥有明显异常注释；
- Level 1 不能通过多数投票重建正确实现；
- Level 3/4 不能把最高指标设置为唯一正确候选；
- 公开输出不得打印隐藏规则或答案标签；
- 同一错误家族应能生成新的数值和代码变体，避免记忆答案。

### 11.4 Level 3/4 实施后形成的结构化证据原则

Level 3 和 Level 4 的完整实现进一步说明：高层审计题不能主要依赖说明文字，也不能只把
一张“看起来可疑”的结果表交给学生。题目必须把科学协议和治理过程编码为可独立重建的
结构化证据。

Level 3 至少应分离四层材料：

1. `planned_runs` 冻结预期的实验单位、配对关系、方法和运行总体；
2. `runs` 保留实际观测、是否纳入以及排除理由；
3. `aggregate` 必须能够从声明纳入的运行中精确重算；
4. 结构化 `claim` 必须声明范围、实际意义阈值和具体 evidence run IDs，并与论文文字使用
   同一个 Claim-ID。

公开检查可以证明 schema、算术和引用关系完整，却不能代替科学判断。隐藏审计必须分别检查
完整调参预算签名、信息条件、预登记排除规则以及每个 `(pair_id, method)` 的真实观测数量。
随机种子、epoch 和 checkpoint 不能因为数量多就自动成为独立科学样本；两组边际区间是否
重叠也不能单独作为显著性判断。

Level 4 应至少提供冻结协议、按时间排序的 Agent events、审批决策、完整 run ledger 和从
ledger 重建的 report manifest。协议条款应使用稳定 clause ID；event 应记录 evidence refs、
decision basis 和 details，并明确哪些字段允许 JSON `null` 或 CSV 空值。

治理违规应由正面证据证明。例如，审批错误必须能指出批准状态、决定时间、精确 scope 或
limits 与行动冲突，不能仅因为某条日志没有 approval ID 就推断现实中从未批准。资源使用应
从 ledger 累加，报告应从完整 ledger 重建。受保护证据违规则应表现为 protected event ID
明确流入后续 adaptive action；单纯的事后报告或讨论不应被误报为自适应。

这些原则对应程序员角色的进一步变化：人不仅要“看结果是否合理”，还要在框架设计阶段
规定什么证据必须被记录、什么关系必须可重建、什么情况下证据不足，以及哪些改变需要在
执行前获得批准。

## 12. 学生应提交什么

每个 Level 的答案至少包含：

1. 唯一可信候选；
2. 对其他候选的精确错误定位；
3. 对应公式、规范、配置键、run_id 或 event_id；
4. 为什么错误仍能正常运行或产生合理结果；
5. 错误到最终结果的因果链；
6. 最小反例、审计断言或最小补跑实验；
7. 不改变冻结协议的修复方式；
8. 对结论强度的校准；
9. 对正确行为的误报控制。

仅选中字母但没有证据，不视为掌握能力。

## 13. 建议评分

每级 100 分：

| 维度 | 分值 |
|---|---:|
| 选出唯一可信候选 | 20 |
| 精确定位与引用证据 | 25 |
| 因果影响解释 | 20 |
| 最小反例、审计断言或补跑方案 | 20 |
| 控制误报 | 10 |
| 表达清晰与结论强度校准 | 5 |

建议达到 80 分再进入下一级。选中字母但没有证据时，得分上限应低于及格线。

## 14. 如何迁移到一篇新论文

### 第一步：冻结论文主张

记录论文声称改善了什么、在哪些数据和指标上改善，以及哪些条件必须保持相同。

### 第二步：建立四层规范

- Level 1：公式、shape、算子和梯度不变量；
- Level 2：样本身份、split、状态恢复和评估协议；
- Level 3：baseline、预算、seed、统计和允许主张；
- Level 4：Agent 权限、审批、预算和停止条件。

### 第三步：建立可信参考

参考不能只来自源码。应交叉使用论文、官方实现、数据说明、单元测试和可手算的小样本；若
论文与源码冲突，必须记录冲突并明确训练题采用的判定规范。

### 第四步：设计最小且高危的语义或协议偏差

优先选择：

- 改动极小；
- shape 和接口不变；
- 输出看似合理；
- 会改变科学语义；
- 存在快速、确定的证明。

“最小”的形态随 Level 改变：Level 1 是局部代码微变异，Level 2 是最小数据流或状态偏差，
Level 3 是单一比较或报告规则违规，Level 4 是一次未经授权或无法审计的行动。

### 第五步：验证题目而不是只验证答案

- 五份候选公开检查全部通过；
- 正确候选通过全部隐藏规则；
- 每份错误候选稳定失败目标规则；
- 无答案泄漏；
- 重复运行结果一致；
- 学生能够从提供材料独立推出结论。

## 15. 这套框架最终训练的能力

完成四级训练后，学习者应能够：

- 阅读 Agent 生成的复杂框架，而不是被“代码量很大”压倒；
- 从论文主张反向建立可检查的不变量；
- 在完整训练前用小实验发现高危错误；
- 追踪数据、模型、状态、指标和结论的证据链；
- 拒绝不公平但结果漂亮的实验；
- 识别 Agent 在长程任务中的权限漂移和协议漂移；
- 对 Agent 输出作出有依据的批准、阻止或请求补证决定；
- 把人类精力集中在决定研究是否可信的环节。

最终原则是：

> 不因为 Agent 很强而逐行监视它，也不因为代码能运行而盲目信任它；用冻结规范、自动
> 不变量、完整账本和人工科学判断建立可验证的信任。

## 16. 与当前 LLM4SBR 训练的关系

LLM4SBR v2 已成为这套通用框架的第一份完整、可运行实例。旧版训练主要覆盖局部算法审计；
v2 已实现五候选盲审、公开检查、隔离答案和隐藏探针，并按以下抽象能力组织：

- 论文核心公式、张量语义和梯度契约用于 Level 1；
- 会话数据、跨模态身份、训练状态和评估协议用于 Level 2；
- baseline 条件、重复实验、统计证据和允许主张用于 Level 3；
- Agent 的授权范围、资源边界、实验记录和受保护证据使用用于 Level 4。

实际实现进一步采用：Level 3 的结构化 experiment config、planned/observed runs、可重算
aggregate 和 evidence-linked claim；Level 4 的稳定协议条款、事件与审批链、资源 ledger、
report manifest 和 protected-evidence flow。公开总入口只报告四级中性 PASS/FAIL，包外隐藏
入口验证每级恰好一份可信候选和四种单一目标错误，而不向学习者输出答案映射。

后续迁移到其他论文时，应保留四级能力结构和证据要求，但替换领域公式、数据契约、指标
和具体治理边界。

# Source-Blind Paper Learning and Paper-Only Reproduction Protocol

> 给 Codex/AI 的项目级指令：默认先在隔离作者源码、候选代码、答案和隐藏探针的条件下学习机器学习/深度学习论文，建立独立的论文审计基准，再进入研究代码四级训练；只有在用户明确选择时，才进一步生成 paper-constrained clean-room reproduction 蓝图。

## 0. 如何使用本文件

### 0.1 先选择模式

本协议有两个互斥模式。用户没有明确指定时，默认使用 **模式 A**。

| 模式 | 目的 | 可读取材料 | 默认输出 | 是否生成实现 |
|---|---|---|---|---|
| **A：Source-Blind Pre-Audit Learning（默认）** | 在看代码前学懂论文，形成独立审计基准，为 Level 1–4 做准备 | 论文、附录、supplement、数据说明、本协议 | `PAPER_STUDY_GUIDE.md` | 不生成源码；允许公式、tensor contract、系统图和解释性伪代码 |
| **B：Paper-Only Clean-Room Reproduction（扩展）** | 在确实没有作者源码时，形成论文约束下的工程复现蓝图 | 与模式 A 相同，可额外查阅经批准的官方框架/数据文档 | `PAPER_REPRODUCTION_GUIDE.md` | 只规划 clean-room 实现；实际写代码需用户另行授权 |

无论选择哪种模式，都不能把第三方实现、Agent 推测或框架默认值冒充作者事实。

### 0.2 推荐目录隔离

1. 新建独立的论文学习目录。
2. 只放入论文 PDF、supplementary material、用户已有的数据说明和本协议。
3. 不把作者源码、训练候选、答案目录、hidden probes 或旧的题目分析放入该目录。
4. 可将本文件复制并重命名为 `AGENTS.md`，使 Codex 自动读取；若保留原文件名，在 Prompt 中明确要求遵循它。
5. 输出指南写在当前隔离目录，不能写入或覆盖原论文、作者源码和训练包。

### 0.3 模式 A 的推荐 Prompt

~~~text
请严格遵循 PAPER_ONLY_REPRODUCTION_PROTOCOL.md，并使用模式 A：Source-Blind Pre-Audit Learning。

本项目可能存在作者源码和四级审计题，但本轮禁止访问。你只能读取当前隔离目录中的论文、附录、补充材料、数据说明和协议文件；不得搜索、读取、引用或推断作者源码、第三方实现、A–E 候选、答案映射、DO_NOT_OPEN_UNTIL_FINISHED/ 或 hidden probes。

请以具备基础 Python 和机器学习知识的 CSE 本科生能够理解的详细程度，在本轮把全部五个 Unit、工程化大框架、PRE_AUDIT_BASELINE 和四级训练衔接说明写入同一个 PAPER_STUDY_GUIDE.md。目标是在首次查看代码之前建立独立的论文数学契约、数据与评估协议、证据边界和未知项清单。不要生成项目源码，不要判断任何候选，不要把伪代码或 [A] 描述成作者实现。对话框只返回简短完成状态和文件链接。
~~~

### 0.4 模式 B 的推荐 Prompt

~~~text
请严格遵循 PAPER_ONLY_REPRODUCTION_PROTOCOL.md，并使用模式 B：Paper-Only Clean-Room Reproduction。这里只有论文和补充材料，没有作者源码。请在本轮把全部五个 Unit 写入同一个 PAPER_REPRODUCTION_GUIDE.md；在 Unit 1 与 Unit 2 之间建立完整的工程化大框架，并给出 paper-constrained clean-room 复现蓝图。所有非论文选择标为 [A]，不得搜索或复制第三方实现。对话框只返回简短完成状态和文件链接。
~~~

本协议适用于：

- 作者源码存在，但学习者希望先隔离代码、独立学懂论文后再做审计训练；
- 作者没有公开源码，需要建立 paper-constrained clean-room reproduction；
- 学习者需要详细、清晰、少跳步的解释和完整工程心智模型；
- 用户希望一次获得完整课程地图，而不是进行长期逐页问答。

本协议不适用于：

- 已经进入逐文件源码对照、缺陷定位或候选判题阶段；
- 要求精确复刻作者未公开的程序行为；
- 只需要论文摘要、文献综述或翻译；
- 希望 Agent 直接透露四级训练答案或替学习者完成审计。

## 1. 核心使命与边界

目标是在一个任务回合内生成一份可作为后续代码审计独立基准的 Markdown 学习指南；模式 B 再在此基础上增加 clean-room reproduction 蓝图。指南必须满足：

1. 学习者能够快速建立论文的完整心理模型。
2. 在任何组件级代码构建和讲解前，学习者先获得完整、论文特定的工程化大框架。
3. 论文中的关键概念、公式、图表和实验得到详细解释。
4. 每个关键公式都被翻译成明确的 tensor contract。
5. 数据、模型、损失、训练和评估被组织成可审计的工程蓝图。
6. 论文没有公开的工程细节被显式标记为假设，而不是补写成作者事实。
7. 每个 Unit 都列出需要注意的地方、常见误解和静默语义错误。
8. 模式 A 最终产物能够作为进入 Level 1–3 的论文基准；模式 B 还能指导从零开始的 PyTorch clean-room reproduction。
9. Unit 1-5 的全部教学正文集中写在同一个文件中；聊天仅承担进度通知和文件交付，不承担正文展示。

核心原则：

> 快速学习意味着减少来回导航、重复阅读和碎片化问答，不意味着减少解释、隐藏中间步骤或把五个 Unit 压成提纲。

默认认知顺序必须是：

~~~text
论文系统图：作者的方法为什么成立、各机制做什么
→ 工程系统图：整套程序怎样分层、怎样运行、各模块怎样交接
→ 组件放大图：Unit 2–5 分别建立什么审计 contract；模式 B 再说明怎样实现和验证
~~~

“60 秒全景”或 Unit 1 中的一条模型数据流不能替代工程系统图。工程系统图必须在 Unit 2 的数据组件细节之前完整出现。

模式 A 允许完成的是：

> 只依据论文公开材料建立任务、公式、tensor、数据、训练、评估和 claim 的独立契约，并明确哪些内容无法在看代码前确定。

模式 A 不能声称：

- “这就是作者源码的结构或行为”；
- “某个候选一定正确或错误”；
- “论文未规定的实现选择只有一种”；
- “指南中的伪代码可以替代后续代码审计”。

模式 B 在没有作者源码时允许完成的是：

> 在明确假设 A1...An 下，满足论文公开数学规格、数据约束和实验协议的一份可运行、可测试、可审计的复现。

模式 B 不能声称：

- “这就是作者的实际实现”；
- “该代码与作者源码完全一致”；
- “论文没写，但作者一定这样处理”；
- “指标接近证明实现路径相同”；
- “指标不一致证明我们的代码一定有 bug”。

Unit 2–5 在模式 A 中用于建立可审计 contract，在模式 B 中进一步形成论文约束下的合理复现蓝图；两种模式都不是猜中作者私有实现。

## 2. 单轮单文件交付合同

### 2.1 “同一任务回合生成一个完整指南文件”的精确定义

Codex 在完成材料检查后，必须在同一个任务回合中把 Unit 1、2、3、4、5 的完整内容写入同一个 Markdown 文件。无论论文长短、内容是否能装进一条聊天消息，这条规则都不变：

~~~text
检查论文与补充材料
→ 静默规划五个 Unit 和组件前置的工程化大框架
→ 选择一个指南文件路径
→ 把五个 Unit、工程化大框架及配套正文写入同一个文件
→ 重新打开并验证文件完整性
→ 对话框只报告完成状态并提供文件链接
→ 不在 Unit 之间等待用户确认
~~~

不得采用以下流程：

~~~text
只给课程目录
→ 只讲 Unit 1
→ 要求用户回复“继续”
→ 后续几轮才给 Unit 2-5
~~~

输出路径与冲突处理规则：

1. 用户指定了文件名或路径时，以该位置为首选；用户未指定时，模式 A 使用当前项目根目录的 `PAPER_STUDY_GUIDE.md`，模式 B 使用 `PAPER_REPRODUCTION_GUIDE.md`。
2. 首选路径不存在时，直接在该位置创建指南。
3. 首选路径已经存在时，无论它来自默认值还是用户指定值，都必须先读取，不能盲目覆盖。
4. 现有文件明确属于同一篇论文和同一指南任务时，可以谨慎更新；兼容的用户增补必须保留。
5. 现有文件属于其他论文、用途不明，或保留用户改动会破坏本协议的完整结构时，不得覆盖或强行合并；应在同一目录选择新的无冲突文件名，并在最终回复中链接实际使用的路径。
6. 备用名在模式 A 采用 `<paper-slug>_STUDY_GUIDE.md`，模式 B 采用 `<paper-slug>_REPRODUCTION_GUIDE.md`。`paper-slug` 从论文 PDF 文件名去掉扩展名后得到：将连续空白替换为 `-`，移除路径分隔符和文件系统不安全字符；结果为空时使用 `paper`。若备用名仍存在，依次追加 `-2`、`-3`，直到得到未占用路径。
7. Unit 1–5、位于 Unit 1 与 Unit 2 之间的“工程化大框架”、统一假设账本、`PRE_AUDIT_BASELINE`、四级训练衔接、工程文件树、实现顺序和结论边界必须集中在本任务实际生成或更新的这一个指南文件中。
8. 不得把五个 Unit 拆成五个文件，也不得把部分 Unit 留在聊天框、其余 Unit 放入文件。
9. “单文件”只约束学习指南的交付；模式 A 不创建项目源码，模式 B 规划出的未来复现源码、测试、配置和报告仍应采用合理的模块化文件结构。

首次完整文件中：

- 五个 Unit 都必须出现；
- Unit 1 与 Unit 2 之间必须出现独立的“工程化大框架（组件实现前置章）”，但它不编号为 Unit 6；
- 五个 Unit 都必须是实质内容，而不是一句话摘要；
- 必须包含 `PRE_AUDIT_BASELINE`，将论文契约、推导、假设和未知项集中整理；
- 必须说明这些材料分别如何支持 Level 1、Level 2、Level 3，以及为什么 Level 4 还需要额外的冻结治理协议；
- 不得使用 `TODO`、`TBD`、“后续补充”或空白模板代替内容；
- 不得先做能力诊断；
- 不得要求用户先自行阅读若干页；
- 不得在 Unit 之间暂停；
- 用户后续追问是可选支持，不是取得剩余 Unit 的条件。

### 2.2 对话框输出与写入失败的处理

工作过程中的 commentary 只能给出简短进度，例如“正在阅读论文”“正在写入并验证指南”。最终聊天回复只允许包含：

- 是否成功生成或更新指南；
- 指南的可点击文件链接；
- 模式 A 必要时说明“这是 source-blind 论文学习基准，没有读取作者源码或训练候选”；
- 模式 B 必要时说明“这是 paper-constrained reproduction，不代表作者私有实现”。

最终聊天回复不得包含 Unit 正文、Unit 标题列表、公式推导、worked example、详细目录或指南摘录。文件已经承载完整内容，不要在聊天中再复制一份。

写完后必须重新打开指南，确认：文件非空；Unit 1–5 各出现一次且没有编号为 `## Unit 6` 的第六单元；Unit 1 与 Unit 2 之间存在完整的工程化大框架；各 Unit 必需结构、统一假设账本、`PRE_AUDIT_BASELINE`、四级训练衔接和工程蓝图存在；最终链接指向实际生成的文件。

如果环境不能写入目标位置，不得退化为在聊天框粘贴五个 Unit，也不得静默拆成多轮或多个文件。应明确报告文件写入阻塞，并请求用户提供一个可写的项目或 artifact 路径。只有完成单文件写入与复核，才算完成交付。

### 2.3 不设置学习者问题和阶段门

首次指南中不得出现：

- 自查题；
- 阅读理解题；
- 参考答案；
- 小测验；
- 闭卷问题；
- “请你先计算/回答”；
- 单元完成 Gate；
- 通过后才能查看下一 Unit 的机制。

可以出现：

- 已完整求解的 worked example；
- tensor shape 示例；
- 算法伪代码；
- 工程测试设计；
- 常见 bug 与最小检测方法；
- 可选的后续追问入口。
- 非阻塞的“进入 Level 1 前准备度清单”，例如能否解释关键公式、shape、mask、target、数据切分和论文未知项；它不得包含候选答案，也不能成为获取剩余 Unit 的条件。

工程测试是验证复现代码的工具，不是学习者考试，也不能阻塞五个 Unit 的展示。

## 3. Source-blind 证据与正确性规则

### 3.1 四种标签

所有会影响实现的技术结论应使用以下标签：

- `[P] Paper`：论文正文、公式、图、表或补充材料明确给出的内容。
- `[A] Assumption`：论文没有唯一规定，为了形成学习基准或构建复现而选择的可替换假设。
- `[E] Experiment`：已经实际运行得到的测试输出、测量或实验结果。
- `[I] Inference`：根据 `[P]`、`[A]` 或 `[E]` 得出的解释和判断。

`[A]` 不是论文证据。它的作用是让无法从论文确定的实现选择保持可见、可替换、可做敏感性实验。

所有会影响数值、语义、随机性、兼容性或评估结果的非论文选择都必须拥有唯一的 `A#`。不能以“只是框架默认值”“影响应该很小”或“这是标准做法”为理由省略标记。

只有论文显式陈述的内容才能标为 `[P]`。由公式、shape、上下文或常识推导出的结果即使看起来唯一，也仍应标为 `[I]`，并列出它所依赖的 `[P]`。

模式 A 不得读取或使用 `[C] Code` 证据；模式 B 没有作者代码 oracle。标准 PyTorch API、库文档、第三方实现或用户后来写出的复现代码，都不等于作者源码。

模式 A 通常不应出现 `[E]`，因为本阶段不运行实现。论文表格中的结果仍是论文报告的 `[P]`，不是本地实际运行的 `[E]`。

### 3.2 四层正确性

指南必须区分四层正确性；模式 A 主要建立第 1 层和第 3 层的审计基准，模式 B 才能规划第 2–4 层的验证：

1. **论文规格一致性**：实现是否满足论文明确公式、文字和 shape 约束。
2. **内部工程正确性**：单元测试、不变量、手算 fixture 和梯度检查是否通过。
3. **复现协议完整性**：数据、假设、随机性、训练和评估规则是否完整公开。
4. **经验结果相容性**：在声明的协议下，结果是否接近论文表格、趋势或消融结论。

前三层通过，不自动证明第四层；结果接近，也不证明代码结构与作者相同。

模式 A 推荐结论措辞：

~~~text
这是仅依据论文公开材料建立的 source-blind 审计基准。它没有读取作者源码、训练候选、答案或隐藏探针；其中 [P] 是论文事实，[I] 是可追溯推导，[A] 是可替换假设。后续代码必须逐项接受该基准的独立审计，指南本身不预判任何候选。
~~~

模式 B 推荐结论措辞：

~~~text
这是一个在假设 A1...An 下，满足论文公开规格并通过已实际运行且记录为 [E] 的内部测试的 clean-room reproduction。由于作者未公开源码或完整工程协议，与作者实际实现的一致性无法验证。
~~~

### 3.3 论文内部矛盾

如果正文、公式、图或实验设置互相不一致：

1. 精确记录各处 `[P]`；
2. 不静默选择“看起来正确”的版本；
3. 说明每种解释会怎样改变 shape、代码或结果；
4. 模式 A 将各解释并列保存在不确定性账本，不选择唯一“正确实现”；
5. 模式 B 才选择一个 provisional `[A]` 作为首个可运行版本，并将其他解释保留为命名配置或敏感性实验；
6. 在指南或报告中明确当前模式和选择，不冒充作者事实。

## 4. 交付前如何阅读论文

在开始写五个 Unit 之前，Codex 必须先完成以下只读工作：

1. 确认当前模式和允许读取的根目录；模式 A 先验证作者源码、候选、答案和隐藏探针均不在读取范围内。
2. 识别主论文、附录、supplementary material 和勘误说明。
3. 获取 PDF 总页数、章节结构、公式编号、图表编号和实验部分位置。
4. 对公式密集页、架构图和关键表格进行视觉检查，不能只依赖文本抽取。
5. 从摘要、引言、方法、实验和结论建立完整因果链。
6. 列出论文明确规定的内容和未公开的实现细节。
7. 找出实现所需的最低前置知识，例如 embedding、GNN、Transformer、对比学习或概率模型。
8. 为缺失前置知识准备嵌入式最小补充课，不另开第六个 Unit。
9. 选一个贯穿五个 Unit 的小型非对称例子，减少重复解释成本。
10. 静默确定 Unit 3 与 Unit 4 的具体边界，使其匹配当前论文结构。
11. 根据论文方法和实验协议，先规划一份论文特定的工程分层、模块依赖、训练链、推理/评估链和跨层接口；它将在 Unit 1 后、Unit 2 前完整交付。

如果论文 PDF 不可读取、缺页或关键公式无法辨认，可以向用户请求重新提供材料。这是少数允许阻止完整交付的情况。

如果只是论文没有写清工程细节，不得因此停止；必须采用第 10 节的假设协议继续。

## 5. 固定五 Unit 结构

五个 Unit 的职责固定，但名称应根据论文内容具体化。Unit 1 建立论文地图；随后用一个不计入 Unit 数量的强制前置章建立工程系统地图；Unit 2–5 在模式 A 中把论文转化为可审计 contract，在模式 B 中进一步转化为 paper-constrained reproduction 蓝图。

## Unit 1：论文全景、问题与完整数学地图

### 本 Unit 的目的

让初学者先获得整篇论文的完整心理模型，知道作者试图解决什么、各组件为什么存在、信息如何流动，以及论文证据能够支持到什么程度。

### 必须详细覆盖

1. **任务定义**

## Unit 3：核心表示机制 A 的数学与 tensor 审计契约

### 本 Unit 的目的

### 必须详细覆盖

   - 它解决 Unit 1 中的哪个信息缺口。

2. **最低前置知识桥接**
   - 只讲当前论文真正使用的机制；

   - 参数 shape；
   - 矩阵乘、逐元素乘、gather、scatter、softmax 或归一化轴；
   - 输出代表什么；
   - loss 如何通过该模块传播梯度。

4. **完整 worked example**
   - 使用不对称、小规模、可以手算的数据；
   - 展示每个关键中间量；
   - 说明结果怎样体现论文机制；

   - 简洁但完整的伪代码或接口骨架。

6. **论文缺失项与可替换实现**
   - 初始化；

   - 边界行为；
   - 层数和隐藏维度；

   - shape 与 finite 检查；

## Unit 4：任务头、可选第二机制与端到端语义契约

   - causal、padding、attention 或 validity mask；

5. **PyTorch 模块树**
   - `model.py` 中各子模块的组合；

~~~text
tests/test_core_b.py

fixtures/model_tiny.*
configs/model_baseline.*
~~~

### Unit 4 常见误解与易错点

- 把 padded 最后一列当成最后有效位置；
- mask 的 `True/False` 语义与框架 API 相反；
- 输入 ID 和 target class 使用不同空间却未转换；
- candidates 多一行或少一行，shape 正常但类别整体错位；
- 手动 softmax 后再传入交叉熵；
- 论文要求共享参数但代码复制了参数值，或论文未规定却擅自 tying；
- train/eval 状态与 autograd 状态混淆；
- 输出排序正确但 loss 使用了错误列；
- 根据个人熟悉的架构补齐论文空白却没有 `[A]` 标记。

## Unit 5：损失、训练、评估与科学证据边界

### 本 Unit 的目的

将 Unit 2–4 连接成完整的训练与评估协议，并区分算法正确性、流水线完整性、比较公平性和论文 claim。模式 A 的目标是冻结审计基准；模式 B 才进一步设计能够逐层建立可信度的复现实验。两种模式都不能用结果接近替代语义证据。

### 必须详细覆盖

1. **所有训练目标**
   - 主损失、辅助损失和正则项；
   - 每个 loss 的输入、target、shape 和 batch reduction；
   - loss 权重放置位置；
   - sampling、augmentation 或 latent variable；
   - 每条 loss 路径更新哪些参数。

2. **训练状态机**
   - `zero_grad → forward → loss → backward → step`；
   - optimizer、scheduler、gradient clipping 和 mixed precision；
   - `train()`、`eval()` 与 `no_grad()`；
   - checkpoint 保存内容和恢复行为；
   - validation、early stopping 和 model selection。

3. **初始化与随机性**
   - 论文明确的初始化和 seed；
   - 未规定时的 provisional `[A]`；
   - Python、NumPy、PyTorch 和数据采样随机性；
   - 多次运行怎样报告均值与方差。

4. **评估协议**
   - metric 的精确定义；
   - 分母、rank、阈值和 tie-breaking；
   - full-catalog 与 sampled evaluation；
   - filtering、special IDs 和 candidate leakage；
   - validation 调参与 test 最终报告的隔离。

5. **由小到大的验证阶梯**
   - 静态 shape 和 fixture；
   - 单组件前向与梯度；
   - tiny batch overfit；
   - 小数据 smoke training；
   - 论文主要指标的 bounded reproduction；
   - 消融与敏感性实验；
   - 资源足够时再尝试完整表格。

6. **实验结果怎样解释**
   - 哪些结果支持实现内部正确；
   - 哪些结果只说明经验相容；
   - 哪些差异可能来自数据版本、随机性或 `[A]`；
   - 不用性能下降单独证明某一行代码有 bug；
   - 不用一次接近的结果宣称作者级精确复现。

7. **最终实现顺序**
   - 先数据 contract；
   - 再 Unit 3；
   - 再 Unit 4；
   - 再 losses 和 metrics；
   - 最后训练与实验；
   - 每一步都先固定 fixture 和可观察中间量。

### Unit 5 的工程产物蓝图（模式 B）

模式 A 只交付 loss/metric 定义、训练状态机、实验单位、比较条件和 claim 边界，不创建下列源码文件。模式 B 可以规划：

~~~text
src/losses.py
src/metrics.py
src/train.py
src/evaluate.py
tests/test_losses.py
tests/test_metrics.py
tests/test_tiny_overfit.py
configs/train_baseline.*
reports/reproduction_report.md
reports/ablations.md
~~~

### Unit 5 常见误解与易错点

- loss 使用 probabilities 而 API 期望 logits；
- batch mean、sum 和 masked mean 混淆；
- 正负样本或候选类别错位；
- 辅助损失权重乘错位置；
- `zero_grad()`、`backward()` 或 `step()` 顺序错误；
- validation 仍启用随机增强或 dropout；
- 使用 test set 选择超参数；
- metric 的 rank 从 0/1 开始处理错误；
- full-catalog 与 sampled-candidate 指标直接比较；
- 只报告最好 seed；
- 训练失败时直接大范围调参，而没有先做 tiny overfit；
- 将结果接近写成“已复刻作者实现”。

## 五个 Unit 之后的强制产物：PRE_AUDIT_BASELINE

无论使用模式 A 还是模式 B，指南都必须在五个 Unit 后给出一个集中、可追溯的 `PRE_AUDIT_BASELINE`。它不是候选答案，也不能包含候选字母、源码位置或 hidden rule；它只记录学习者在看代码前能够从论文建立的判断标准。

至少包含：

1. **论文 claim 表**：claim ID、论文位置、适用数据/设置、证据类型、允许的结论范围。
2. **算法契约表**：公式 ID、输入输出语义、shape、axis、mask、index/label 空间、loss 与 gradient path。
3. **数据与评估协议表**：独立实体、切分顺序、样本构造、候选集合、metric 分母、checkpoint/test 边界。
4. **实验比较表**：baseline、信息条件、调参预算、实验单位、配对方式、排除规则和报告统计量。
5. **不确定性账本**：每个 `[I]` 和 `[A]` 的依据、备选解释、可能影响，以及进入代码阶段后需要核对的证据。
6. **禁止预判清单**：仅凭论文不能确定的类名、文件结构、API 默认值、具体实现顺序、候选正确性和作者私有行为。
7. **Level 1 准备度清单**：学习者能否在不看代码的情况下解释核心公式、关键轴、mask、target、loss 和 gradient contract。

四级衔接必须写清：

| 后续 Level | 从本指南带入的基准 | 进入该 Level 后新增的证据 |
|---|---|---|
| Level 1：算法语义 | 公式、tensor、axis、mask、target、gradient contract | 候选实现、最小 tensor、property/gradient probe |
| Level 2：流水线完整性 | 实体身份、切分、样本、metric、checkpoint 边界 | 多文件代码、运行 artifact、跨阶段 lineage |
| Level 3：科学有效性 | baseline 条件、预算、实验单位、排除规则、claim scope | planned/observed runs、aggregate、structured claim |
| Level 4：Agent 治理 | 论文只能提供部分资源与实验背景 | 额外冻结的审批、预算、停止、记录和 protected-evidence 协议；不得从论文凭空推导 |

模式 A 完成后应停在这里。除非用户明确宣布进入四级训练，否则 Agent 不得自动打开源码、候选或任何隔离材料。

## 6. 每个 Unit 的详细度合同

每个 Unit 必须按照下列顺序组织，不能只列关键词：

1. **在整套系统中的位置**：上游、下游和本 Unit 输出。
2. **要解决的问题**：没有它会缺什么。
3. **通俗机制**：为初学者解释直觉和术语。
4. **论文规格**：引用章节、公式、图或表。
5. **数学与 tensor contract**：语义、shape、轴、参数和梯度。
6. **完整 worked example**：展示中间过程和结果。
7. **工程映射**：模式 A 给出模块职责、接口和解释性伪代码；模式 B 可以进一步给出未来文件蓝图。
8. **假设与备选方案**：论文没有规定的所有关键选择。
9. **注意事项与易错点**：错误、后果、检测方法和修正 contract。
10. **本 Unit 的产出**：模式 A 为审计 contract、worked example 和未知项；模式 B 再增加未来代码、fixture、测试或报告蓝图。

每个 Unit 的可观察最低产物是：

- 至少一条本 Unit 数据流；
- 至少一张相关公式组的 tensor-contract 表；
- 至少一个完整求解的非对称 worked example；
- 至少一个工程接口、模块树或关键伪代码块；
- 一组明确的 `[A]` 或“本 Unit 无关键缺失项”的说明；
- 一组包含后果与检测方法的易错点。

这些是讲义组成部分，不是学习者题目或阶段验收。

### 6.1 “详细”的最低标准

详细不等于重复同一句话。详细意味着：

- 缩写第一次出现时解释；
- 术语先讲直觉，再给正式定义；
- 公式先讲目的，再逐个解释符号；
- 单样本公式必须给出 batch 版本；
- 每次矩阵运算都说明关键 shape；
- 每次 softmax、sum、mean 或 normalize 都说明轴；
- 每个 target 都说明类别空间和索引；
- 每个 mask 都说明 `True/False` 或 `0/1` 的含义；
- 每个随机操作都说明训练/评估行为；
- 每个 provisional 选择都标记 `[A]`；
- 每个常见错误都说明为什么 shape 可能仍然正常；
- PyTorch 伪代码必须能看出模块边界和数据流。

### 6.2 避免五个 Unit 重复

- Unit 1 只建立全景、方法公式第一遍和一句式前置直觉，不重复 Unit 3-5 的实现细节。
- 工程化大框架只建立完整软件系统的分层、调用链、依赖、状态与跨层接口，不代替组件内部算法。
- Unit 2 只建立数据协议，不提前写完整模型。
- Unit 3 深入上游核心表示机制。
- Unit 4 深入任务头、端到端集成，以及论文确有的可选第二机制。
- Unit 5 深入 loss、训练、评估和实验证据。
- 后面的 Unit 可以引用前面的符号与结论，但必须补充新的工程层信息。

## 7. 初学者解释标准

默认学习者会基础 Python，但不假设已经熟悉论文使用的所有模型。

遇到新机制时采用：

~~~text
为什么需要它
→ 一个具体小例子
→ 它对 tensor 做了什么
→ 论文公式
→ PyTorch 映射
→ 容易写错在哪里
~~~

必须避免：

- 用另一个未定义术语解释当前术语；
- 直接说“这是标准做法”而不说明标准做法是什么；
- 跳过从单样本到 batch 的变化；
- 只贴公式不讲分母、轴和索引；
- 只贴大段代码不解释数据流；
- 因追求简洁而省略假设和边界行为。

如果论文依赖 GNN、Transformer、contrastive learning、diffusion、variational inference 等学习者可能没学过的内容，在相关 Unit 内加入“本文最小前置知识”小节。只讲足以理解和实现本文的部分，不增加新 Unit。

### 7.1 常见机制的最小桥接清单

只使用与当前论文相关的行，不机械讲全表；被论文实际使用的行必须落到 tensor 和 PyTorch 操作，而不能只给一句定义。

| 机制 | 至少讲清的实现概念 |
|---|---|
| GNN | 节点/边/特征、邻居、message/aggregate/update、方向与自环、层数与 hop、归一化轴、gather/scatter 或 sparse aggregation、图与样本 batch 的区别 |
| Transformer | token 表示、Q/K/V、attention score、softmax 轴、position、padding/causal mask、残差与归一化、选择哪个输出表示 |
| Contrastive learning | anchor/positive/negative、正负身份来源、similarity、temperature、candidate logits、target 位置、InfoNCE 与 CE 的关系 |
| CNN | channel、kernel、stride、padding、receptive field、输入布局、卷积后 shape |
| Diffusion | forward noising、timestep、noise schedule、预测目标、reverse sampling、训练与采样差异 |
| VAE | encoder 参数、latent distribution、reparameterization、reconstruction、KL、训练与生成路径 |

如果论文使用其他机制，按同样标准提取“对象、数据流、关键轴、参数、梯度与 PyTorch 操作”，而不是增加一门完整基础课程。

## 8. 公式到 PyTorch 的统一翻译格式

每个共享同一数据流的公式组至少提供一张 contract 表：

| 字段 | 必须说明 |
|---|---|
| 论文位置 | 章节、公式、图或表 |
| 目标 | 该公式组解决什么 |
| 输入语义 | 每个输入代表什么 |
| 单样本 shape | 论文符号的原始维度 |
| batch shape | PyTorch 中加入 batch 后的维度 |
| dtype/device | long、float、bool 以及设备 |
| 运算 | matmul、gather、scatter、softmax、mask 等 |
| 归一化轴 | 分母或 softmax 沿哪个集合 |
| 参数 | 可学习参数的 shape 和初始化来源 |
| 输出 | 输出语义与 shape |
| 梯度 | 哪些参数可以从哪条 loss 路径更新 |
| `[A]` | 论文未规定的实现选择 |
| 易错点 | 最可能的静默语义错误 |

对复杂公式可以再提供等价伪代码，但不能用伪代码替代数学解释。

## 9. 易错点与静默 bug 的写法

每个真正影响实现的易错点应说明四件事：

1. **错误做法**：具体会写成什么。
2. **为什么诱人**：来自广播、API 默认值、符号歧义还是惯性实现。
3. **可观察后果**：shape 报错、数值变化、信息泄漏、梯度中断或指标偏差。
4. **最小检测方法**：怎样用一个不对称 fixture、断言或短实验发现它。

优先覆盖：

- 一个会直接 shape 报错的问题；
- 一个 shape 正常但语义错误的问题；
- 一个方向、索引或 target 对齐错误；
- 一个 normalization/softmax 轴错误；
- 一个 padding、mask 或候选集合错误；
- 一个梯度或 train/eval 状态错误；
- 一个数据泄漏或评估协议错误；
- 一个将 `[A]` 冒充 `[P]` 的证据错误。

不要只写“注意维度”“可能出错”或“检查数据”。必须写明错在哪里、为什么错、怎样发现。

## 10. 论文缺失细节的处理协议

遇到论文没有公开的工程细节时，依次执行：

1. 搜索论文正文、脚注、附录、supplement 和实验设置，确认真的缺失。
2. 写出论文已经给出的相邻约束 `[P]`。
3. 列出会影响实现的合理选项。
4. 判断是否能由 shape、公式或数据统计唯一推出；即使可以，推导结果也标为 `[I]`，不能升级为 `[P]`。
5. 如果不能唯一推出，模式 A 保留所有高影响合理解释，不强行冻结唯一答案；模式 B 才选择最简单、常见且与 `[P]` 不冲突的 provisional baseline。
6. 将每个选择标为 `[A]`，给出理由和潜在影响；模式 A 还要注明“进入代码阶段后需要核对的证据”。
7. 模式 B 将高影响备选做成配置，而不是散落在代码里。
8. 设计小型 sensitivity experiment 或 invariant test。
9. 在结果报告中重新列出所有关键 `[A]`。

统一假设账本格式：

| ID | 未规定事项 | 已知 `[P]` | 合理选项 | 采用的 `[A]` | 理由 | 潜在影响 | 检测/敏感性实验 |
|---|---|---|---|---|---|---|---|

以下内容尤其容易缺失，必须主动查找和记录：

- 数据版本和预处理顺序；
- ID、padding、mask 和 label 空间；
- 初始化；
- normalization、activation 和 residual；
- sampling 分布及有无放回；
- augmentations 的作用范围和概率；
- candidate set；
- loss reduction；
- optimizer、scheduler 和 gradient clipping；
- early stopping 与 checkpoint；
- metric 细节和 tie-breaking；
- seed、运行次数和报告统计量。

## 12. 从论文学习进入代码工作的顺序

模式 A 完成五个 Unit 和 `PRE_AUDIT_BASELINE` 后应停止。学习者明确开始四级训练时，按 Level 1 → 2 → 3 → 4 顺序使用论文基准审计训练材料；不能让 Agent 自动替学习者判题。

只有模式 B 或用户另行授权实际复现时，工程实现才按以下依赖顺序执行：

1. 先按前置大框架建立项目骨架、模块职责和接口占位；此时不填充未经推导的组件算法。
2. 建立 `PAPER_SPEC.md` 和 `ASSUMPTIONS.md`。
3. 创建一个完全可手算的 synthetic fixture。
4. 实现 Unit 2 数据协议并验证中间对象。
5. 实现 Unit 3 核心机制 A。
6. 实现 Unit 4 的任务头、论文确有时的可选第二机制，以及端到端 forward。
7. 实现 Unit 5 losses 和 metrics。
8. 检查 finite forward、finite gradients 和目标参数梯度。
9. 在单个 tiny batch 上过拟合。
10. 运行小数据 smoke experiment。
11. 再尝试论文主实验、消融和敏感性分析。

不要直接从下载完整数据、长时间训练和调参开始。指标不能替代组件语义测试。

### 12.1 复现证据阶梯

| 层级 | 证据 | 能说明什么 |
|---|---|---|
| R0 | shape、dtype、finite | 程序基本可运行 |
| R1 | 手算 fixture 与不变量 | 关键组件符合声明的语义 |
| R2 | finite gradients、tiny overfit | 优化路径基本连通 |
| R3 | 小数据稳定训练 | 训练协议可以工作 |
| R4 | 多 seed 主结果与趋势 | 与论文经验结果的相容程度 |
| R5 | 消融与敏感性 | 关键设计和 `[A]` 的影响 |

这些层级是工程证据组织方式，不是学习者阶段门，也不能延迟五个 Unit 的首次交付。

## 13. 工具、网络与代码来源边界

- 将论文及补充材料视为主要规格来源。
- 模式 A 只能读取用户明确提供的隔离学习目录。即使工作区中存在作者源码、训练包或答案，也不得搜索文件名、执行全文检索、读取 Git 历史或通过工具间接取得其内容。
- 模式 A 不得浏览或搜索作者源码、第三方实现、候选代码、题解、博客中的实现分析或本训练包的公开/隐藏测试。
- 模式 B 也不得因为没有作者源码就静默搜索并复制第三方仓库。若用户明确允许查找非官方实现，必须标记来源，且不能把它提升为作者 code oracle。
- 模式 B 可以查阅 PyTorch 官方文档、数据集官方说明和指标定义；模式 A 只有在论文概念无法可靠解释且用户允许时才可查阅，并必须作为外部文档单独引用，不能标成论文 `[P]`。
- 框架文档可以证明某个 API 的默认行为；决定在本复现中采用该默认行为仍然是一个 `[A]`。
- 模式 A 不运行代码、数据处理或训练；模式 B 不自动下载大型数据集或启动长训练。
- 不为了匹配旧论文版本而破坏用户现有 Python 环境；需要时建立隔离环境。
- 优先使用当前可用 PyTorch 和纯 Python/PyTorch reference。
- 模式 B 的随机实验必须记录 seed、设备、版本、命令和配置。
- 任何尚未运行的测试计划都不能标为 `[E]`，也不能写成“测试通过”；只有实际命令、输出和环境已记录时才能声称通过。

## 14. 生成指南文件的固定结构

模式 A 默认生成 `PAPER_STUDY_GUIDE.md`，结构如下：

~~~markdown
# [论文名]：Source-Blind 论文学习与代码审计准备指南

## 使用边界
- 当前模式与允许读取的材料
- 明确隔离的源码、候选、答案和隐藏材料
- source-blind 正确性声明

## 60 秒全景
- 任务
- 核心思想
- 完整数据流
- 五 Unit 关系

## Unit 1：[论文全景具体标题]
[详细内容]

## 工程化大框架（组件实现前置章）
[论文特定的工程分层与依赖图、训练链、推理/评估链、公式到模块映射、跨层接口表、状态生命周期、实现顺序和最小纵向切片]

## Unit 2：[数据与审计规格具体标题]
[详细内容]

## Unit 3：[核心机制 A 的数学与 tensor 契约标题]
[详细内容]

## Unit 4：[任务头/可选第二机制/端到端集成的具体标题]
[详细内容]

## Unit 5：[损失、训练、评估与证据边界标题]
[详细内容]

## 统一假设账本
[所有关键 A1...An]

## PRE_AUDIT_BASELINE
[claim、算法、数据/评估、实验比较、不确定性、禁止预判和 Level 1 准备度清单]

## 四级训练衔接
[Level 1–4 分别带入什么论文基准、进入后新增什么证据]

## 论文理解结论边界
[允许声称什么、看代码前不能声称什么]

## 可选后续支持
[回答论文疑问；只有用户明确宣布进入训练后，才指导打开对应学员材料]
~~~

模式 B 默认生成 `PAPER_REPRODUCTION_GUIDE.md`。它使用同样的五 Unit、工程化大框架、假设账本和 `PRE_AUDIT_BASELINE`，并在末尾增加：

~~~markdown
## 工程实施清单（承接前置大框架）
[汇总未来模块、tests、fixtures、configs、reports 和分阶段验证命令]

## 复现结论边界
[允许声称什么、不能声称什么]
~~~

每个 `[详细内容]` 必须满足第 6 节的十项结构，不能保留占位符。“工程化大框架”必须满足第 5 节前置章的全部最低可观察产物，不能退化为一段总述或通用目录。模式 A 中出现的工程文件名和伪代码只能是解释性映射，不能形成源码产物或被称为作者设计。

模式 A 最终聊天只负责交付文件，可以使用以下最小格式：

~~~markdown
已生成完整的 paper-only 学习与工程复现指南：

[PAPER_REPRODUCTION_GUIDE.md](/absolute/path/to/PAPER_REPRODUCTION_GUIDE.md)

五个 Unit、组件前置的工程化大框架、假设账本和工程实现顺序均在文件中；对话框不重复正文。
~~~

模式 B 如需说明正确性边界，只再增加一句：该指南描述的是论文约束下的 clean-room reproduction，无法证明与作者未公开实现完全一致。

## 15. 后续互动规则

首次任务回合生成的单一指南文件已经包含全部五个 Unit 和组件前置的工程化大框架。用户后续可以自由选择：

- 对某个概念要求更通俗的解释；
- 深挖某个公式；
- 在模式 B 中要求生成项目 scaffold；
- 在模式 B 中开始实现 Unit 2–5；
- 明确宣布进入 Level 1，并让 Agent 只提供不泄题的学习路径；
- 提交代码进行检查；
- 运行 tiny fixture 或训练实验；
- 修改某个 `[A]` 并比较结果。

回答追问时：

- 先直接回答问题；
- 说明它属于哪个 Unit；
- 保持 `[P]/[A]/[E]/[I]` 区分；
- 不要求用户重新阅读整份指南；
- 不把后续追问变成新的 Unit；
- 不声称作者使用了当前复现选择。
- 模式 A 在用户明确宣布进入四级训练前继续保持 source quarantine；进入训练后也不得读取 `DO_NOT_OPEN_UNTIL_FINISHED/`、答案 manifest 或 hidden probes。
- 使用 `PRE_AUDIT_BASELINE` 审计候选时，必须保持 `[P]`、`[I]` 和 `[A]` 的层级；候选与某个 `[A]` 不同，不足以判定候选错误。

如果用户随后明确要求 actual clean-room implementation，切换到模式 B 或新的实现任务，正常执行实现、测试与验证。若用户进入四级训练，Agent 只能担任学习向导，不能替学习者打开答案或直接给出候选结论。

## 16. 常见失败模式与纠正

| 失败行为 | 正确行为 |
|---|---|
| 在聊天框直接输出五个 Unit | 把全部正文写入同一个 Markdown 指南，聊天只给完成状态和链接 |
| 论文较短时直接在聊天框讲完 | 无论长短都生成单一指南文件 |
| 把五个 Unit 拆成多个讲义文件 | Unit 1-5、工程化大框架及配套账本集中写入同一个指南文件 |
| 为控制篇幅只给五个标题 | 在同轮创建完整 Markdown 指南并验证内容 |
| 只讲 Unit 1 后等待“继续” | 在单一指南文件中一次性交付 Unit 1-5 |
| 模式 A 发现工作区有作者源码就顺手读取 | 保持 source quarantine，只读取隔离学习目录中的论文材料 |
| 模式 A 搜索候选、答案或 hidden tests 来“提高准确性” | 停止搜索；只建立论文基准，不预判训练题 |
| 把某个 `[A]` 当作之后判错的依据 | 只有 `[P]` 或可追溯且必要的 `[I]` 才能建立强制契约；`[A]` 保留为待核对分支 |
| 模式 A 直接生成完整项目源码 | 只生成学习指南、contract、系统图和解释性伪代码；实际实现需要切换模式或另行授权 |
| 先要求用户阅读论文 | AI 先预读，再提供组织后的讲义 |
| 先做 PyTorch 能力诊断 | 在相关 Unit 内补充最低前置知识 |
| Unit 1 讲过模型数据流后直接进入 Unit 2 | 在两者之间完整交付论文特定的工程化大框架 |
| 把工程文件树和实现顺序留到 Unit 5 之后才讲 | 组件细节前先讲分层、职责、依赖、训练/推理链、状态和接口，末尾只做实施汇总 |
| 用通用项目目录冒充工程架构 | 将当前论文的公式、数据对象、分支、loss 和评估规则逐一映射到模块与调用链 |
| 把训练和推理画成同一条链 | 分开说明两条生命周期，并标出推理时消失的 loss、梯度、更新与随机增强 |
| 每个 Unit 后添加小测 | 删除小测，改用完整 worked example |
| 用 Gate 控制下一 Unit | 五个 Unit 在同一指南文件中无条件全部展示 |
| 论文没写就采用框架默认值 | 标记 `[A]`，解释理由和影响 |
| 把标准做法称为作者做法 | 写成“模式 B 的复现采用”或“模式 A 的 provisional `[A]`” |
| 把第三方实现当官方源码 | 只作为外部参考并明确来源 |
| 只讲公式，不讲 batch | 同时给单样本和 batch contract |
| 只讲代码，不讲语义 | 先说明问题、机制和不变量 |
| 只列“注意维度” | 写出错误、后果和最小检测方法 |
| 一开始追完整论文指标 | 先 fixture、梯度、tiny overfit 和小数据 |
| 指标接近就宣布精确复现 | 只声明经验结果相容 |

## 17. 行为压力场景

### 场景 A：论文很长，完整指南可能过长

正确行为：

- 仍覆盖全部五个 Unit；
- 删除重复背景和无关 related work；
- 保留公式 contract、工程映射、`[A]` 和易错点；
- 在同轮把完整内容写入一个 Markdown 文件；论文较短时也执行同样规则；
- 聊天只返回完成状态和该文件的链接；
- 不停在 Unit 1 或 Unit 3 等用户继续。

### 场景 B：论文缺少 padding、初始化或 sampling 细节

正确行为：

- 先确认正文和附录均未说明；
- 列出合理选项；
- 模式 A 保留高影响备选，不冻结唯一答案；模式 B 才选择 provisional `[A]`；
- 说明影响并提供配置/敏感性实验；
- 不写“作者采用”。

### 场景 C：学习者是初学者

正确行为：

- 解释术语和最低前置知识；
- 使用贯穿全文的小例子；
- 展开 shape、轴、mask、target 和梯度；
- 保持同一任务回合、单一指南文件的五 Unit 交付；
- 不用诊断题延迟讲义。

### 场景 D：论文只有一个核心模型分支

正确行为：

- Unit 3 负责表示构建；
- Unit 4 负责预测头、解码、候选输出与端到端集成；
- 不制造不存在的第二分支。

### 场景 E：论文实验结果无法复现

正确行为：

- 该场景只适用于模式 B 或后续实现任务；模式 A 不运行复现实验；
- 先定位证据阶梯中最早失败的层级；
- 检查数据版本、协议和高影响 `[A]`；
- 做 bounded sensitivity experiments；
- 将结果记录为 `[E]`；
- 不自动断言论文错误或实现错误。

### 场景 F：用户要求“完全复刻作者实现”

正确行为：

- 明确说明没有作者源码和完整工程协议时无法验证实现身份；
- 提供 paper-constrained reproduction；
- 列出无法确定的部分；
- 不用自信措辞掩盖信息缺口。

### 场景 G：作者源码存在，但用户要先学论文

正确行为：

- 使用模式 A，并把源码、候选、答案和隐藏测试排除在允许读取范围外；
- 不因本机能够访问仓库就读取或搜索它；
- 输出 `PAPER_STUDY_GUIDE.md` 和 `PRE_AUDIT_BASELINE`；
- 用户明确进入 Level 1 后才接触学员可见候选，并继续隔离答案材料。

### 场景 H：论文很简单，只有一个 encoder 和一个 CE loss

正确行为：

- 仍在 Unit 1 与 Unit 2 之间提供工程化大框架；
- 可以缩短不存在层的说明，但不能省略训练链、推理/评估链、模块依赖、接口交接和实现顺序；
- 明确哪些层被合并到同一文件，以及为什么这种合并对小论文足够；
- 不因组件少就从论文摘要直接跳到 `Dataset` 或 `forward` 代码。

## 19. 本协议的完成标准

一次合格的首次交付应使学习者通过一个完整 Markdown 文件、且无需来回问答便获得：

- 整篇论文的问题、动机、架构和公式地图；
- 一份先于组件细节出现的工程系统地图，包括分层、文件职责、依赖方向、训练链、推理/评估链、状态生命周期和跨层接口；
- 从原始数据到模型输入的可审计协议；
- 一个工程深度明确的核心表示 Unit，以及一个任务头/端到端集成 Unit；论文确有第二机制时才在后者展开；
- 完整 loss、训练、评估和实验设计；
- 关键 tensor shapes、axes、masks、targets 和 gradient paths；
- 每个 Unit 的易错点及最小检测方法；
- 一份集中、透明的 `[A]` 假设账本；
- 一份 `PRE_AUDIT_BASELINE`，集中保存论文 claim、算法契约、数据/评估协议、实验比较条件、未知项和禁止预判项；
- 一张 Level 1–4 衔接表，说明论文基准与进入各级后新增证据的边界；
- 模式 B 额外获得模块化 PyTorch 文件树、实现顺序和从 fixture 到 bounded reproduction 的证据路线；
- 一个准确的结论边界：模式 A 是未读取代码的论文审计基准，模式 B 是论文约束下的复现蓝图；两者都不是作者私有代码的重建。

最终原则：

> 默认先在 source quarantine 下把五个详细 Unit、工程化大框架、`PRE_AUDIT_BASELINE` 和四级训练衔接写入一个完整 Markdown 文件，让学习者在看代码前形成独立判断基准。只有明确选择模式 B 时才进一步规划 clean-room reproduction。所有未知都分层记录为 `[I]` 或 `[A]`，任何指南内容都不能预判候选或冒充作者源码事实。

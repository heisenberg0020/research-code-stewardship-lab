# LLM4SBR 四级研究审计训练设计规格

日期：2026-07-21  
状态：概念设计已获用户确认；等待书面规格确认后实施

## 1. 目标

在不破坏现有 `LLM4SBR_code_judgement_training` 的前提下，新建
`LLM4SBR_research_audit_training_v2`，把训练从局部 PyTorch 判错扩展为四级研究审计：

1. 算法语义正确性；
2. 数据、训练、评估流水线正确性；
3. 实验公平性与科学主张有效性；
4. Coding Agent 的权限、协议和实验治理合规性。

每一级都是一套独立盲题，包含 A-E 五份外观和接口一致的候选材料，其中仅一份完全
可信，其余四份分别包含一种集中、可解释、不会被简单编译检查暴露的错误。

训练的最终能力不是“猜中正确字母”，而是能够提交可审计证据：错误定位、违反的协议
或公式、因果影响、最小反例，以及安全修复或正确治理动作。

## 2. 非目标

- 不运行完整 MovieLens-1M 或 Beauty 长训练。
- 不下载 BERT、Qwen 或其他远程模型。
- 不以获得论文指标或 SOTA 为验收条件。
- 不把代码改动比例当作错误严重度指标。
- 不要求学生使用答案目录中的探针完成题目。
- 不修改原论文、原仓库和旧版训练包。

`FRAMEWORK_OVERVIEW.md` 使用 `docs/FOUR_LEVEL_RESEARCH_CODE_AUDIT_OVERVIEW.md` 的通用
内容作为训练入口，介绍 Coding Agent 能力边界、程序员角色、四级能力映射和通用错误分类；
它不得包含本论文各候选的正确字母、具体突变位置或隐藏探针答案。

四级不是线性难度排序。每道题按“首先被破坏的契约”确定唯一主 Level：局部公式或算子
语义为 Level 1；自动流水线中的身份、数据流、状态或评估契约为 Level 2；比较设计、证据
汇总或主张推断为 Level 3；未经授权的 Agent 自适应行动或治理违规为 Level 4。下游后果
可以跨层，但不得因此为同一候选设置多个主要错误。

候选正确字母在四级之间不重复使用固定模式。学生侧脚本不得导入答案目录，也不得通过
文件名、注释、不同代码风格或异常文件长度暗示正确答案。

## 4. 共用数据与运行约束

### 4.1 合成 SBR 数据

使用固定随机种子生成小型确定性数据：

- 原始 session 具有 `session_id`、按时间排序的 item 序列和语义向量；
- 从原始 session 切出 prefix-target 样本；
- item 0 固定为 padding；
- train/validation/test 具有冻结的 session 级时间边界；
- 数据足够小，CPU 数秒内完成 Level 1/2 检查；
- 数据刻意包含变长 session、重复 item、冷启动边界和非均匀候选集。

### 4.2 可重复性

- 所有随机源显式使用 seed；
- 同一公开检查连续执行两次结果一致；
- 不依赖网络、GPU、外部服务或未声明数据；
- 使用当前 `ml` Conda 环境的 PyTorch；Level 3/4 仅依赖 Python 标准库和 CSV/JSON。

### 4.3 公开检查与隐藏答案

公开检查只证明材料完整、可解析、可运行和结果有限，不告诉学生哪个结果符合规范。
隐藏答案包含：

- 正确候选；
- 每个错误候选的证据链；
- 能稳定区分错误的探针；
- 修复建议；
- 标准评分示例。

## 5. Level 1：算法语义审计

### 5.1 能力目标

把论文公式 (4)-(15) 映射为张量语义，识别 shape 正确、可反向传播但算法不正确的
实现，并用最小张量反例证明判断。Level 1 专门训练“微型但高危”的语义错误：错误
原则上只占一个逻辑行、一个参数、一个常数或一个梯度操作，代码改动比例可以低于
0.1%，但足以使论文中的数学目标、优化路径或预测概率失真。

Level 1 不使用一眼即可从变量名看出的长短期分支整体接反，也不使用明显的循环删除、
shape mismatch、编译错误或大段错误重写。判断必须依赖算子语义、数学不变量或梯度探针。

### 5.2 候选错误类型

四份错误候选分别包含：

1. **归一化维度错误**：余弦相似度使用 `dim=0` 而不是 embedding 维 `dim=-1`；
   shape 和数值均正常，但每个 session 的语义表示依赖同 batch 的其他 session；
2. **mask 填充值错误**：softmax 前将 padding logit 填成 `0.0` 而不是接近负无穷；
   padding 获得合法概率质量，global 表示产生隐蔽的长度偏置；
3. **辅助损失梯度被切断**：在组合后的 DirectAU 辅助损失上附加 `.detach()`；
   forward loss 数值和日志完全不变，但 alignment/uniformity 不再产生任何训练梯度；
4. **交叉熵输入语义错误**：先对 logits 做 softmax，再传入 `cross_entropy`；
   代码仍可训练，预测类别甚至可能相同，但损失和梯度不再等价于论文定义的 logits
   交叉熵。

每个目标错误只能占一条逻辑语句或一个算子参数。隐藏探针必须证明其科学影响，而不仅是
指出文本差异：分别检查 batch-composition invariance、padding invariance、辅助梯度非零性
和大间隔 logits 下的 loss/梯度等价性。

现有旧题只作为接口和论文映射来源，不直接复用原四种错误。v2 中五份候选采用独立但
行为等价的无害实现变体，例如局部变量拆分、等价索引写法和顺序无关的校验表达式；所有
候选保持相同接口、近似行数和统一注释风格。这样正确候选不会简单表现为“在所有差异点都
与多数文件一致”，学生不能通过机械 diff 或多数投票代替语义审计。

### 5.3 学生交付

- 唯一正确候选；
- 四份错误的函数级定位；
- 对应论文公式；
- 为什么接口和训练仍能运行；
- 每份错误的最小反例或不变量测试；
- 预期对 loss、梯度或预测的影响。

### 5.4 引导强度

只保留论文页码、公式编号、输入输出 shape 和任务背景，不给出函数级映射，不直接列出
四个错误类别或不变量。旧版的明显测试方向全部移入答案解析，以避免学生按提示逐项搜索。
公开 smoke test 必须让五份候选都通过，且至少有两份错误候选与正确候选拥有相同的预测
argmax 或相同的记录 loss，防止仅凭单次数值大小猜答案。

## 6. Level 2：流水线完整性审计

### 6.1 能力目标

跨数据预处理、shuffle、训练、checkpoint 和评估文件追踪样本身份与实验状态，识别
“模型正常训练且指标合理”背后的流水线错误。

### 6.2 冻结协议

- 原始 session 必须整体属于一个 split；同一 session 的 prefix 不得跨 split；
- 输入、target、long embedding、short embedding 必须共享同一 permutation；
- validation 选择 checkpoint，test 只允许最终一次评估；
- HR/MRR/NDCG 对每个样本均产生一个值，未命中必须记 0；
- item 0 不属于候选，target 与去除 padding 后的 logits 下标严格对应。

### 6.3 五份候选

一份完全遵守协议，四份各含一种错误：

1. **prefix 泄漏**：先切 prefix 再随机分 split，同一原始 session 出现在 train/test；
2. **模态错位**：shuffle session 和 target，却未同步重排长短期 embedding；
3. **指标偏高**：对 Top-K 内未命中的部分样本没有补 0，导致条件均值；
4. **test 选模**：每个 epoch 在 test 上评估并选择最佳 checkpoint。

所有候选必须完成训练、保存 checkpoint 并输出格式一致的合理指标。公开 smoke test 对五份
候选全部通过；隐藏探针分别检查 session_id 集合、样本联合指纹、逐样本指标长度和评估
事件日志。

### 6.4 学生交付

- 正确 pipeline；
- 每个错误的跨文件数据流证据；
- 能自动复现问题的审计脚本或断言；
- 指标偏差方向与可能影响；
- 若现有证据不能确定偏差方向，明确标为未知；
- 不改变冻结协议的修复方案。

## 7. Level 3：科学有效性审计

### 7.1 能力目标

在代码均可运行、结果表均漂亮的情况下，判断比较是否公平、统计是否完整，以及论文主张
是否被证据支持。

### 7.2 每份 dossier 的组成

- `METHODS.md`：数据、候选集、特征和调参协议；
- `experiment_config.json`：冻结配置与搜索预算；
- `runs.csv`：所有 seed 和运行状态；
- `aggregate.csv`：均值、标准差、样本数和区间；
- `claim.md`：拟写入论文的结论；
- `provenance.json`：数据版本、代码版本和运行来源。

`REVIEW_CRITERIA.md` 还必须预先冻结实验单位、模型间的配对关系、重复结构、允许的统计方法
和主张阈值。随机 seed 只描述训练随机性，不自动构成彼此独立的科学样本；候选若使用同一
数据划分下的重复运行，必须按冻结协议解释可推广范围。

### 7.3 五份候选

一份 dossier 的比较与主张均可信，四份分别包含：

1. **baseline 不公平**：新模型获得更大调参预算或更优超参数，baseline 使用默认配置；
2. **候选集/信息不一致**：新模型使用全候选或额外语义信息，而 baseline 的设置不同；
3. **选择性报告**：只报告新模型最好的 seed，却汇总 baseline 的全部 seed；
4. **统计单位与推断错误**：将同一运行中的 epoch/checkpoint 等相关观测当作独立重复，
   或破坏预先规定的配对单位，再据此声称显著、稳定、普遍优越。

数据数值由固定脚本生成，确保错误不是靠肉眼记忆某个数字，而是由 dossier 内部约束冲突
暴露。公开 validator 只检查 schema；隐藏 validator 检查预算、候选集、运行完整性、
实验单位、配对差异和主张允许范围。不能仅凭两组边际区间是否重叠判断显著性。

### 7.4 学生交付

- 唯一可接受 dossier；
- 对其余 dossier 作 `reject` 或 `revise` 决策；
- 引用具体配置项、运行行号或主张句子；
- 给出可允许的最强结论；
- 说明需要补跑的最小实验，而不是笼统要求“更多实验”。

## 8. Level 4：Agent 权限与科学实验治理审计

### 8.1 能力目标

审查 Agent 是否遵守冻结协议和审批边界。重点不是结果好坏，而是每个行动是否有权限、
是否可追踪，以及最终报告是否忠实反映实验账本。

### 8.2 冻结协议

协议明确：

- 允许的超参数及范围；
- 最大运行数、GPU 小时和失败重试次数；
- validation/test 使用边界；
- 数据、模型、loss、指标和候选集的禁止修改项；
- 必须请求人工批准的操作；
- 失败与负结果必须保留；
- 超预算或关键歧义时必须停止。

训练材料同时声明：所提供的审批记录、Agent 事件、实验账本和允许的通信通道在题目时间窗
内是封闭、完整且按时间排序的，所有有效批准都有可引用 ID。若无法作此完整性声明，则
候选必须提供能够正面证明行动与批准冲突的证据；不能只用“未找到批准”推断“从未批准”。

### 8.3 每份 Agent run 的组成

- `agent_events.jsonl`：带 event_id、时间、actor、action、reason、scope；
- `approval_log.jsonl`：请求、批准、拒绝和未决事项；
- `experiment_ledger.csv`：包括成功、失败和取消运行；
- `final_report.md`：Agent 汇总结论；
- `workspace_diff.txt`：模拟的最终改动摘要。

### 8.4 五份候选

一份 run 全程合规，四份分别存在：

1. **未经批准修改数据协议**：因结果差而改变 sessionization 或 split；
2. **未经批准扩大资源/搜索空间**：增加 GPU 预算或超出超参数边界；
3. **删除或隐瞒失败实验**：ledger 与报告不一致；
4. **test peeking**：根据 test 指标决定下一次模型或超参数修改。

错误 run 可能得到更好的最终指标；正确 run 可能停止、保留负结果或请求审批。材料设计必须
避免把“最高分”与“最可信”绑定。

### 8.5 学生交付

- 唯一合规 run；
- 对每份 run 给出 `approve`、`block` 或 `needs-human-approval`；
- 引用 event_id 和对应协议条款；
- 说明 Agent 在违规点应采取的正确动作；
- 判断最终报告是否可由 ledger 重建；
- 区分工程失败、科学风险和权限越界。

## 9. 评分与升级门槛

每级 100 分：

- 选出唯一可信候选：20；
- 精确定位与引用证据：25；
- 因果影响解释：20；
- 最小反例、审计断言或补跑方案：20；
- 控制误报、不指控正确行为：10；
- 表达清晰、结论强度校准：5。

建议 80 分解锁下一级；若选中字母正确但没有证据，最高 35 分。Level 4 完成后，学生应能
把同一套审计框架用于真实 Agent 实验，而不依赖预设错误清单。

## 10. 测试驱动实施策略

实施前先创建包级验收测试，并确认在 v2 目录不存在时按预期失败。随后依次实现最小内容
使测试转绿。

### 10.1 包结构测试

- 恰好四级；
- 每级恰好五个匿名候选；
- 每级存在 README、作答纸、公开检查和隔离答案；
- 学生侧文件不包含正确候选标签或答案路径导入。

### 10.2 Level 1/2 行为测试

- 五份候选均能导入或执行；
- 五份候选公开 smoke test 全部通过；
- 正确候选通过全部隐藏不变量；
- 每个错误候选恰好稳定失败其目标隐藏探针；
- 前向、反向、指标和日志均为有限值；
- 连续两次运行结果一致。

Level 1 还必须满足：

- 每个错误候选相对其自身正确语义仅有一个微型突变点；
- 四个突变都保持接口、shape、有限 forward 和可执行 backward；
- 梯度切断候选的记录 loss 与正确语义数值一致，但参数梯度证明确有差异；
- double-softmax 候选的 argmax 可与正确语义一致，但梯度和概率校准探针失败；
- 任意候选不得因为代码结构最接近其余四份而被多数投票唯一识别。

### 10.3 Level 3/4 证据测试

- 所有 dossier/run 公开 schema 检查通过；
- 正确候选通过全部隐藏政策检查；
- 每个错误候选恰好触发其目标科学或治理规则；
- 证据引用的 run_id、event_id 和配置键真实存在；
- 报告中的汇总可从原始 ledger/runs 重算。

### 10.4 答案泄漏测试

- 学生侧不出现 `correct candidate`、`known fault` 等答案措辞；
- 不通过文件大小、排序、额外注释或不同命令暴露正确项；
- 公开输出只显示 PASS/schema-valid，不输出隐藏规则结果；
- 正确字母在四级中采用预先冻结但非规律序列。
- Level 1 对所有候选执行 pairwise diff 特征检查；不得存在“选择各突变点多数写法即可直接
  拼出唯一正确候选”的结构性泄漏。

## 11. 验收标准

只有同时满足以下条件才可宣布完成：

1. 根级完整性测试通过；
2. 四级所有公开检查退出码为 0；
3. 四级答案探针精确识别一份正确和四份目标错误；
4. 所有结果可重复；
5. 学生侧泄漏扫描通过；
6. README 命令从新目录实际执行成功；
7. 原仓库和旧版训练包的原有文件未被修改；
8. 最终交付列出每级能力目标、运行命令和建议完成顺序，但不透露答案。

## 12. 实施边界

- 新建 v2 目录，不覆盖旧版答案或候选文件；
- 文件编辑使用补丁方式；
